## 1. 前言

推理工程的目标是让生成式 AI 模型运行得更快、成本更低、更可靠，进而帮助你打造更优质的产品。

> 💡 **关键**：只有当你的推理工程工作落地到生产环境，并伴随成功 AI 产品的高速增长与流量暴增实现规模化扩展时，这一目标才能真正实现。

### 生产环境的新挑战

在对生产流量进行扩缩容时，你的所有假设都会经受严格检验。

```
影响生产性能的因素：
  - 序列形态
  - 流量模式
  - 用户选择聊天的主题
  - ...一切因素都会影响你在生产环境中观测到的性能表现
```

> ⚠️ **重要**：维护安全、稳健的基础设施，与在 GPU 上优化模型推理是**完全不同的技术领域**。

### 为什么需要生产部署工程？

无论单个实例服务模型的速度有多快、效率有多高，当流量足够大时，服务都会不堪重负。

```
这不是 PyTorch 的问题
也不是 CUDA 的问题
而是基础设施问题，需要截然不同的思维模式与技术方案。
```

生产环境的扩缩容会带来新的复杂挑战：

| 挑战 | 说明 |
|------|------|
| **GPU 获取** | 如何获取足够的 GPU 资源 |
| **流量均衡** | 如何在 GPU 之间均衡流量 |
| **服务停机** | 如何避免服务停机 |
| **成本核算** | 从按百万 token 付费转向直接为基础设施付费后，成本核算会变得复杂 |

> 📌 **重要**：生产环境中的延迟不仅来自预填充（prefill）和解码（decode）阶段。你需要对系统进行**端到端评估**，消除服务器、网络，甚至在你能掌控或影响客户端代码的情况下**客户端本身的低效问题**。

## 2. 容器化（Containerization）

### 基本原理

容器化是将应用程序与其依赖项打包在一起，以标准化生产部署方式的实践。

> 💡 容器将程序转化为可在任意环境运行的打包制品 —— **彻底告别"在我机器上能跑"的问题**。

容器十分轻量，因为它们共享底层主机操作系统内核（此处内核指 Linux 内核，而非 CUDA 内核）。这让容器非常适合打包推理服务。

### 关键术语

| 术语 | 说明 |
|------|------|
| **容器（Container）** | 正在运行的隔离环境，用于隔离应用程序及其依赖项 |
| **镜像（Image）** | 可执行软件包，包含运行软件所需的全部内容 |
| **Dockerfile** | 人类可读的文件，包含创建镜像的规范、机器可解析的指令 |
| **仓库（Registry）** | 用于管理、存储、共享和分发镜像的中央存储库 |

> 📌 **AI 领域最常用的仓库**：Docker Hub —— Docker Hub 之于镜像，就像 Hugging Face 之于模型权重、PyPi 之于 Python 包。

---

### Docker 容器层次结构

Docker 容器由多层构成。你可以使用预配置的基础镜像，在此基础上添加包含文件系统变更的其他层。

```
┌─────────────────────────────────────────────────────────────┐
│                    Docker 容器层次结构                       │
├─────────────────────────────────────────────────────────────┤
│  容器层（Container Layer）                                 │
│  • 运行时创建的薄型临时层                                │
│  • 对运行中容器的所有修改写入该层                        │
│  • 容器终止后数据会丢失                                  │
├─────────────────────────────────────────────────────────────┤
│  附加层（Additional Layers）                              │
│  • 文件系统变更                                          │
│  • 包括依赖项、应用代码和 Dockerfile 指令指定的配置文件   │
├─────────────────────────────────────────────────────────────┤
│  基础镜像（Base Image）                                   │
│  • Ubuntu 等操作系统发行版                                │
│  • 或从仓库获取的更复杂镜像                              │
│  • 基础镜像本身由多层组成                                │
└─────────────────────────────────────────────────────────────┘
```

层分为三类：

| 层类型 | 说明 |
|--------|------|
| **基础镜像（Base Image）** | 可以是 Ubuntu 等操作系统发行版，也可以是从仓库获取的更复杂镜像。基础镜像本身由多层组成 |
| **附加层（Additional Layers）** | 文件系统变更，包括依赖项、应用代码和 Dockerfile 指令指定的配置文件 |
| **容器层（Container Layer）** | 运行时创建的薄型临时层。对运行中容器的所有修改（如创建、更新、删除文件）都会写入该层，容器终止后数据会丢失 |

### 2.1 依赖管理

推理的依赖链冗长且脆弱。构建出可用版本难度很大，因此在变更频发的生态系统中，容器化对保留稳定可用版本至关重要。

#### 镜像构成

镜像针对特定 GPU 架构与模型构建。一个容器包含众多运行时组件：

| 组件 | 说明 |
|------|------|
| **CUDA 工具包版本** | 与技术栈其他部分兼容的 CUDA、cuDNN 和驱动的具体版本 |
| **Python 包** | torch、transformers、diffusers 等依赖项 |
| **推理引擎** | vLLM、SGLang、TensorRT-LLM 或其他推理引擎的版本 |
| **系统包** | ffmpeg 等 Linux 包，在处理音频、图像或视频模型时尤为常见 |

> 💡 **就像背包旅行的徒步者一样，你需要轻装上阵**。用于推理的镜像通常有好几 GB 大小。为实现快速部署与高效运行，仅包含严格必需的依赖项即可。

#### 固定版本（Pinning Versions）

另一个最佳实践是**固定版本**。固定依赖树能保证系统在不同环境中的运行行为一致，支持重复构建出相同结果的镜像。

```dockerfile
# ✅ 推荐：固定精确版本
FROM pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime

RUN pip install torch==2.1.0+cu121 torchvision==0.16.0+cu121
RUN pip install transformers==4.36.0
RUN pip install vllm==0.2.7

# ❌ 不推荐：未固定版本
# RUN pip install transformers  # 可能安装不兼容版本
```

> 📌 **好处**：uv、poetry 或 pip 等工具会在构建镜像时检测版本不兼容并抛出错误。固定版本后，只要镜像成功构建一次，后续都会解析到相同版本的依赖，避免变更导致故障。

#### 新模型发布的挑战

新发布模型时，破坏性变更尤为常见。

```
当 DeepSeek 等模型发布新版本时，
整个推理生态系统都会争相提供首日支持。
```

**应对策略**：

| 阶段 | 策略 |
|------|------|
| **模型发布初期** | 依赖项的夜间构建版或其他开发者预览版 |
| **模型发布数天至数周内** | 往往需要基于稳定版重新构建 |

### 2.2 NIMs（NVIDIA 推理微服务）

NVIDIA 推理微服务（NVIDIA Inference Microservices，NIMs）是针对主流开源模型**预构建的 Docker 容器**。

容器让推理服务的实现具备可移植性。NVIDIA 推出了两类 NIM：

| NIM 类型 | 说明 |
|-----------|------|
| **多 LLM NIM** | 灵活型容器，用于在支持的 GPU 架构上运行一类模型 |
| **特定 LLM NIM** | 针对特定 GPU 上的特定模型优化的引擎，配置为实现最高性能 |

NIM 支持多种常见的模型、GPU 架构、GPU 数量与配置组合。

#### NIM 的使用方式

NIM 与普通容器无异。你可以：

| 使用方式 | 说明 |
|---------|------|
| **作为构建起点** | 在 NIM 基础上定制自己的镜像 |
| **作为参考架构学习** | 学习 NVIDIA 的最佳实践 |
| **直接开箱即用** | 作为推理服务直接使用 |

> 💡 **如果你追求最大程度的可控性，而非现成配置，通常更适合从非定制化的基础镜像自行构建容器，而非改造 NIM**。

## 3. 自动扩缩容（Auto-Scaling）

### 核心目标

自动扩缩容的目标是：**始终拥有足够资源服务所有 incoming 请求，同时满足延迟 SLA，且不浪费资金在闲置 GPU 上**。

```
┌─────────────────────────────────────────────────────────────┐
│           无自动扩缩容 vs 优秀自动扩缩容                      │
├──────────────────────┬──────────────────────────────────────┤
│  无自动扩缩容        │  优秀自动扩缩容                        │
│  • 流量低谷：资源浪费 │  • 资源与需求动态匹配                │
│  • 流量峰值：无法满足 │  • 满足 SLA 的同时避免资源浪费       │
│    SLA               │                                      │
└──────────────────────┴──────────────────────────────────────┘
```

---

### 自动扩缩容系统架构

自动扩缩容系统基于 Kubernetes（开源容器编排系统），搭配集群级别的计算资源分配与回收系统运行。

**Kubernetes 基础**：

```
Kubernetes 将一组硬件资源组合为集群。

集群包含两类组件：
  ├── 控制平面（Control Plane）
  │     负责路由与扩缩容决策
  │
  └── 工作平面（Worker Plane）
        运行实际的容器化应用
```

> 📌 Kubernetes 集群可运行多个模型的多个副本。但如何决定每个模型的副本数量？**除非流量异常稳定，否则几乎不存在能完美匹配需求的固定副本数**。

---

### 扩缩容决策依据

自动扩缩容是动态调整集群中指定模型副本数量的实践。

扩缩容决策有两种依据：

| 依据 | 说明 | 特点 |
|------|------|------|
| **利用率（Utilization）** | 根据内存使用率、计算使用率等 GPU 利用率信号扩缩容 | 滞后指标 |
| **流量（Traffic）** | 根据系统中处理的请求数量扩缩容 | 可主动执行 |

> 💡 **最佳实践**：利用率与流量并非总是匹配。
>
> **例如**：在 LLM 预填充阶段，少数包含数十万未缓存输入 token 的请求，可能比大量高缓存命中率的小请求产生更高的利用率。
>
> **建议**：结合使用两者，让系统资源与需求匹配。

---

### 自动扩缩容配置参数

设计基于流量的自动扩缩容系统时，需配置五个参数：

| 参数 | 说明 | 建议 |
|------|------|------|
| **最小副本数（Min Replicas）** | 无论流量如何，始终运行的最小副本数量 | 保证基线服务能力 |
| **最大副本数（Max Replicas）** | 高流量时可分配的最大副本数量 | 防止成本失控 |
| **扩缩容窗口（Autoscaling Window）** | 用于测量流量并做出扩缩容决策的滑动时间窗口 | 太短→抖动，太长→响应慢 |
| **缩容延迟（Scale Down Delay）** | 建议缩容后，为应对流量再次突增而等待的时长 | 防止流量波动时过早缩容 |
| **并发目标（Concurrency Target）** | 每个副本可同时处理的请求数量 | 关键性能参数 |

> ⚠️ **配置权衡**：具体配置决定了自动扩缩容系统能否在满足延迟 SLA 的同时避免资源浪费。
>
> **例如**：延长缩容延迟可防止流量波动时过早缩容，但流量真正回落时会产生不必要的开销。

---

### 3.1 并发与批处理大小

要正常运行基于流量的自动扩缩容系统，必须清楚**每个实例能承载的并发流量**。

#### 批处理类型对比

大多数模型推理服务可通过批处理同时处理多个请求。批处理有多种方式：

```
┌─────────────────────────────────────────────────────────────┐
│                   三种批处理策略对比                         │
├──────────────┬──────────────────┬──────────────────────────┤
│  静态批处理   │  动态批处理       │  连续批处理                │
│  （Static）  │  （Dynamic）     │  （Continuous）           │
│              │                   │                           │
│ 服务器等待    │  服务器等待       │  服务器持续运行推理        │
│ 批处理满后    │  批处理满或      │  有空位时立即接入新请求    │
│ 再开始推理    │  达到超时时间后   │  （Token 级别批处理）     │
│              │  再开始推理      │                           │
│              │                   │                           │
│ ❌ 早期请求   │  ⭐ 平衡          │  ✅ 最小延迟              │
│    等待时间过长│                   │                           │
└──────────────┴──────────────────┴──────────────────────────┘
```

**各推理引擎的批处理能力**：

| 推理引擎 | 批处理支持 |
|---------|-----------|
| **vLLM** | ✅ 连续批处理（Continuous Batching）|
| **SGLang** | ✅ 连续批处理 |
| **TensorRT-LLM** | ✅ 飞行中批处理（Inflight Batching，即连续批处理）|

> 📌 TensorRT-LLM 称为"飞行中批处理"，在 token 级别对请求进行批处理。相比静态批处理，这种方式能**最小化延迟**。

#### 批处理大小：延迟与吞吐量的权衡

批处理大小是**延迟与吞吐量的权衡**。

```
增大批大小 → 提升总吞吐量，但会增加每个用户的延迟
减小批大小 → 降低延迟，但总吞吐量下降
```

> 💡 **建议**：需测试多个批大小的性能，找到适合模型、实例、延迟目标与预算的方案。

#### 配置一致性

该参数在自动扩缩容配置层通过**并发目标**控制，在副本层通过**批大小**控制，两者需保持一致。

```
当所有活跃副本达到最大并发数时 → 自动扩缩容系统会启动更多副本
若大量副本运行半满批次 → 则应缩容
```

### 3.2 冷启动

**冷启动**是启动模型新副本所需的时间。

> ⚠️ **自动扩缩容系统的整体性能取决于冷启动速度**。若副本启动速度过慢，就无法放心缩容，导致过度配置资源。

#### 冷启动流程与优化

冷启动时间受四个因素影响，每个因素都需要单独优化：

```
┌─────────────────────────────────────────────────────────────┐
│                    冷启动流程                                │
├──────────┬──────────────────────────────────────────────────┤
│ 阶段 1    │  GPU 获取                                     │
│          │  集群添加所需 GPU 并分配给模型的速度               │
│          │  → 优化：使用热节点池（预配置 GPU 的节点池）    │
├──────────┼──────────────────────────────────────────────────┤
│ 阶段 2    │  镜像加载                                     │
│          │  将容器镜像加载到新获取实例的速度                │
│          │  → 优化：减小镜像体积，仅包含必需步骤与依赖项    │
├──────────┼──────────────────────────────────────────────────┤
│ 阶段 3    │  模型加载                                     │
│          │  将模型权重加载到容器的速度                      │
│          │  → 优化：模型权重量化，加快加载速度             │
│          │  → 优化：从邻近缓存源加载（高带宽）             │
├──────────┼──────────────────────────────────────────────────┤
│ 阶段 4    │  引擎启动                                     │
│          │  启动推理引擎的速度（包括编译时间）              │
│          │  → 优化：缓存已构建的引擎                       │
└──────────┴──────────────────────────────────────────────────┘
```

#### 各阶段优化详解

**① GPU 获取优化**

除非你有可在模型间灵活调配的**热节点池**，否则 GPU 获取速度主要取决于云服务商。

> 💡 节点启动时间是合同中可协商的条款之一。

**② 镜像与权重加载优化**

镜像与权重加载速度取决于向实例写入数据（通常数百 GB）的速度。

加快加载有两种方式：

| 方法 | 说明 |
|------|------|
| **减小体积** | 仅包含必需步骤与依赖项可减小镜像体积，加快容器构建速度 |
| **提升带宽** | 模型权重量化还能加快冷启动时的加载速度 |

**权重加载位置对带宽影响极大**：

| 加载位置 | 带宽 | 推荐度 |
|---------|------|--------|
| **从 Hugging Face 等第三方加载** | 受其出口带宽限制 | ❌ 不推荐 |
| **从 S3 桶加载** | 引入网络延迟与数据传输成本 | ⚠️ 可接受 |
| **从 GPU 实例物理邻近的缓存源加载** | 需要 GB/s 级带宽 | ✅ 推荐 |

> 💡 **最佳实践**：加载数百亿参数的模型，需要 **GB/s 级带宽**。最佳方式是在同一数据中心内，从 GPU 实例物理邻近的缓存源，通过节点内网络加载。

**③ 引擎启动优化**

vLLM、SGLang 等推理引擎启动速度快。但 TensorRT-LLM 等引擎与 PyTorch 优化模型需要针对特定硬件编译推理引擎，编译过程通常需要**数分钟**。

> 💡 **解决方案**：缓存已构建的引擎可大幅提升冷启动速度。
>
> - TensorRT-LLM 与 PyTorch 均提供镜像缓存机制
> - **关键**：必须在与引擎构建环境**完全相同的 GPU 类型、CUDA 版本与软件依赖**的实例中加载缓存引擎，才能正常运行

---

### 3.3 路由、负载均衡与队列

当多个副本在线时，系统需要决定将哪些请求发送到哪些副本。

#### 组件分类

做出这些决策的组件分为两类：

| 组件类型 | 决策层级 | 回答问题 |
|---------|---------|---------|
| **路由器（Routers）** | 请求级别 | "这个请求该发往哪里" |
| **负载均衡器（Load Balancers）** | 系统级别 | "这个请求可以发往哪里" |

> 📌 复杂系统中并非只有一个路由器和一个负载均衡器。路由贯穿整个技术栈，关键节点插入负载均衡器以保证系统整体性能稳定。

#### 智能路由策略

总体目标是在副本间均匀分配负载。但路由与负载均衡并非简单的"3 个副本、12 个请求，每个副本分配 4 个请求"。

```
每个请求的输入 token 数量可能不同。
若大多数请求有 100 个输入 token，
一个包含 10,000 个 token 的请求会打破简单系统的均衡。
```

某些请求更适合由特定副本处理：

| 路由策略 | 说明 |
|---------|------|
| **KV 缓存感知路由** | 将请求导向已在 KV 缓存中存在匹配前缀的副本 |
| **LoRA 感知路由** | 将请求导向已在内存中加载目标 LoRA 微调权重的副本 |

> 💡 智能请求路由利用推理引擎与 NVIDIA Dynamo 等编排器的信息，根据序列长度、前缀与 LoRA 需求路由请求。

#### 队列系统

仅靠负载均衡与路由不够。当自动扩缩容系统接收的流量超出处理能力时，需要在扩容更多资源或等待现有资源释放期间**暂存请求**。

```
队列是处理这种情况的基础设施基础组件。

标准队列：先进先出（FIFO）的超额请求管理系统

复杂队列逻辑：
  • 优先级队列（高流量场景下付费用户优先于免费用户）
  • 延迟队列（按 SLA 要求延迟处理）
  • 死信队列（处理失败请求的再处理）
```

> ⚠️ **新副本上线后**，需确保队列感知到新副本，避免请求持续等待现有副本。每个新副本激活后，应立即分配队列中流量至并发上限。

---

### 3.4 缩容至零

高级自动扩缩容系统支持**缩容至零机制**：无流量时将活跃副本缩至零，收到流量后自动扩容。

#### 缩容至零的前提

缩容至零依赖两个前提：

| 前提 | 说明 |
|------|------|
| **快速冷启动** | 用户可能实时等待，冷启动必须尽可能快 |
| **稳健队列** | 系统能暂存 incoming 请求，直到副本上线 |

#### 适用场景与不适用场景

即便具备这些能力，缩容至零也并非适用于所有工作负载。

| 场景 | 是否适用 | 说明 |
|------|---------|------|
| **开发环境** | ✅ 适合 | 测试流量突发、首请求延迟无关紧要 |
| **周期性流量应用** | ✅ 适合 | 例如仅在某国工作时间访问的智能体，或设计用于每日批处理任务的离线系统 |
| **对延迟敏感、流量轻且无规律的应用** | ❌ 不适合 | 依赖缩容至零控制成本，通常说明你的 AI 应用尚未准备好专用基础设施，应先使用按 token 付费的 API，直到规模扩大 |

---

### 3.5 组件独立扩缩容

AI 应用越来越多地基于**多模型、多阶段的复合 AI 工作负载**构建，推理工程师需要协调多个步骤完成单个请求。

#### 问题：流水线中各步骤需求不同

这些步骤的硬件需求不同。

```
示例：语音处理流水线

语音活动检测模型 → 转录模型 → LLM 处理 → TTS 模型
      │                    │
      │                    │
   所需 GPU 远弱于     可能需要多 GPU 完整节点
   其处理数据的
```

> 📌 流水线中每个步骤的扩缩容参数也不同。

#### 解决方案：组件独立扩缩容

```
┌─────────────────────────────────────────────────────────────┐
│             组件独立扩缩容                                      │
├─────────────────────────────────────────────────────────────┤
│  语音活动检测模型                         独立扩缩容          │
│      ↓                                                      │
│  转录模型（ASR）                          独立扩缩容          │
│      ↓                                                      │
│  LLM 处理                                  独立扩缩容          │
│      ↓                                                      │
│  TTS 模型                                  独立扩缩容          │
│                                                             │
│  → 为每个模型分配合适资源，实现单独扩缩容                   │
│  → 避免瓶颈与过度配置                                     │
└─────────────────────────────────────────────────────────────┘
```

**关键建议**：

> ⚠️ **所有模型应运行在同一集群**。
>
> - 若集群内消息传递需 10 毫秒，集群间需 50 毫秒
> - 5 步流水线的 40 毫秒差异会占用 1 秒延迟 SLA 的 **20%**

---

## 4. 多云容量管理（Multi-Cloud Capacity）

### 为什么需要多云？

单集群内的自动扩缩容有一定上限。服务全球用户的大规模部署需要分布在全球的数千块 GPU。

```
将多云推理构建为不同云服务商的孤立计算资源很简单，
但这种架构无法灵活使用跨云算力，
在云间迁移工作负载繁琐且易出错。
```

> 💡 **真正的多云推理**需要构建多区域、多服务商的资源整合工具，将不同算力池视为可互换资源。类似于单集群内的 Kubernetes，多云容量管理需要**全局视角**，支持自愈与全局调度。

---

### 多云架构

从单一云的单一集群扩展到多云多集群，需要新的协调层。

```
┌─────────────────────────────────────────────────────────────┐
│                  多云架构                                     │
├──────────────────────┬──────────────────────────────────────┤
│  控制平面               │  工作平面                             │
│  （全局）              │  （区域/集群级）                      │
│                      │                                      │
│  • 处理模型部署        │  • 处理直接推理流量                  │
│  • 全局扩缩容决策      │  • 集群内扩缩容决策                  │
│  • 接收实时事件流      │  • 上报利用率与需求                  │
└──────────────────────┴──────────────────────────────────────┘
```

> 💡 **职责分离的好处**：这种职责分离确保单个工作平面可独立服务流量。若控制平面或任一工作平面故障，其他工作平面不受影响。

---

### 多云推理的优势

运行真正的多云推理可实现：

| 优势 | 说明 |
|------|------|
| **容量** | 整合多家服务商的容量，获得更灵活、更充足的 GPU 资源 |
| **冗余** | 跨服务商拆分推理，抵御服务中断 |
| **延迟** | 在靠近终端用户的位置运行推理，降低网络延迟开销 |
| **合规** | 满足数据主权与其他监管要求 |

---

### 4.1 GPU 获取

提供 GPU 访问的公司众多，主要分为三类：

| 类型 | 代表服务商 | 特点 |
|------|-----------|------|
| **超大规模云服务商（Hyperscalers）** | AWS、GCP、Aliyun 等大型云服务商 | 价格较高，但生态完善 |
| **新型云服务商（Neoclouds）** | CoreWeave、Nebius 等专注 GPU 的云服务商 | GPU 可用性通常更好 |
| **经销商（Resellers）** | SF Compute Company 等二级市场服务商 | 价格可能更优，但可靠性需评估 |

> ⚠️ **首要挑战**：保障算力。获取所需 GPU（尤其是最新硬件）往往难度很大，大型集群更难获取，提供数百节点规模的服务商较少。

#### GPU 获取方式

云 GPU 有三种获取方式：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **预留实例（Reserved）** | 以折扣价按月 / 年预订数百或数千块 GPU | 基线容量 |
| **按需实例（On-Demand）** | 按需要使用单个实例，配额内价格较高 | 中等流量波动 |
| **竞价实例（Spot）** | 折扣价按需实例，可在约定通知期（通常数分钟）内被抢占 | 容错工作负载 |

> 💡 **大规模推理通常混合使用 GPU 资源**：
> - 低成本**预留实例**作为基线
> - **按需实例**与**竞价实例**应对流量峰值
> - 这些 GPU 分布在全球多个集群，靠近终端用户

---

### 4.2 地理感知负载均衡

成功的 AI 应用拥有全球用户。

```
如同单集群需要负载均衡器保证集群内每块 GPU 获得合理流量，
多集群系统需要全局负载均衡器。
```

> ❌ **你不希望**：
> - 用户请求在其他位置有空闲资源时仍在队列中等待
> - 习惯性地将新加坡的请求发送到旧金山服务器

#### 延迟预算考量

```
经验法则：
  请求跨越一个时区需要 5 毫秒。
  → 因此，从纽约到旧金山的数据单向传输需要 15 毫秒。

考虑到延迟预算紧张，
尽可能在靠近终端用户的位置运行工作负载至关重要。
```

---

### 4.3 可靠性构建

> ⚠️ **GPU 在生产环境中的故障率极高**。所有进行过大规模训练的工程师都知道，必须考虑硬件故障的可能性。

#### Llama 3 训练的故障数据

在 Llama 3 论文中，Grattafiori 及其团队披露：

```
在 54 天内使用 16,000 块 GPU 运行时，
共发生 419 次意外中断，
主要原因是硬件故障。
→ 大约每 5 万 GPU 小时出现一次故障。
```

> 💡 **换算**：5 万小时看似很长，但单节点 8 块 GPU 全年推理的 GPU 小时数超过 **7 万**。
>
> **结论**：推理工程师应预期硬件故障。

#### GPU 故障处理

```
┌─────────────────────────────────────────────────────────────┐
│                  GPU 故障处理                                 │
├─────────────────────────────────────────────────────────────┤
│  节点级：                                                      │
│  • 单块 GPU 故障时，节点上其他 GPU 通常会相继故障          │
│    或需要下线维护                                           │
│  • 主动发现故障、隔离节点、重启 Pod                         │
│  • 保证单个集群健康                                          │
│                                                           │
│  多云级：                                                     │
│  • GPU 故障并非唯一导致推理中断的原因                         │
│  • 云服务商有计划内维护与计划外停机                         │
│  • 必须加固基础设施的每一层，实现高可靠性                   │
└─────────────────────────────────────────────────────────────┘
```

#### 多云高可靠性方案

多云推理为高可靠性提供两种新方案：

| 方案 | 说明 | 特点 |
|------|------|------|
| **双活模式（Active-Active）** | 高可用架构，多区域 / 多集群同时服务实时流量。任一平面故障，流量无缝切换至其他平面 | 成本较高，但可用性最高 |
| **主备模式（Active-Passive）** | 故障转移架构，"热备" 集群 / 区域就绪但闲置。主平面故障时，流量切换至备平面 | 成本较低，但切换有延迟 |

> ✅ **单个集群、区域或云服务商故障时，无缝切换至其他工作平面可保证高可靠性与低延迟**。

---

### 4.4 安全与合规

二十多年来，云基础设施一直是安全与合规部门的关注重点。要让 AI 模型支撑关键业务应用，推理必须**安全且合规**。

#### 安全与合规的三大领域

| 领域 | 说明 | 保护措施 |
|------|------|---------|
| **用户数据** | 安全与合规部门要求保护所有数据，包括用户输入与模型输出 | 数据加密、访问控制 |
| **模型权重** | 对于微调或专有模型公司，权重是宝贵的商业机密 | 加密存储、访问审计 |
| **基础设施** | GPU 本身与智能访问都是滥用目标 | 网络隔离、入侵检测 |

#### 提升安全性的实践

**① 不存储用户输入与模型输出**

提升安全性最简单的决策之一是不存储用户输入与模型输出。

> 这可能无法实现 —— 你可能有日志记录要求或用户协议，需要保留使用数据用于未来模型训练 —— 但如果无需保留用户数据，**可减少攻击面**。

**② 遵循容器安全最佳实践**

保护 AI 推理工作负载与相关数据，与保护其他容器化工作负载类似：

```
数据加密 → 传输中加密 + 静态加密
容器安全 → 镜像漏洞扫描 + 最小权限原则
网络与访问控制 → 防火墙 + IAM 策略
工作负载隔离 → 命名空间隔离 + 资源配额
第三方渗透测试验证 → 定期安全审计
```

> 📌 **再经过广泛的第三方渗透测试验证，仍是黄金标准**。

**③ 满足合规要求**

推理工程师越来越需要支持受监管行业与合规严格区域的应用。

多云基础设施的优势之一是，若应用需要满足 **SOC 2 Type II** 认证或 **HIPAA** 等法规，服务商本身也必须合规。

> 💡 **此时，将工作负载迁移至合规服务商非常有用**。

**④ 数据驻留要求**

多集群基础设施的另一个优势是**跨区域运行单个模型**。

部分行业与国家有数据驻留要求，本国用户数据不得在其他国家服务器处理。

```
示例：
  • 在多伦多附近服务商部署一个集群（加拿大数据）
  • 在纽约附近部署另一个集群（美国数据）
  • 可让加拿大数据留在加拿大、美国数据留在美国
  • 同时为该地理区域用户提供最小延迟开销
```

## 5. 测试与部署（Testing & Deployment）

### 测试策略

除了配置推理引擎时进行的副本级测试与基准测试，部署前进行**端到端系统测试**同样重要。

推理测试有多种策略：

| 测试策略 | 说明 | 成本 |
|---------|------|------|
| **手动测试** | 编写脚本（或点击按钮）向推理服务发送模拟流量 | 低 |
| **负载测试** | 自动发送大量流量，测试系统扩缩容与保持性能的能力 | 中 |
| **影子流量（Shadow Traffic）** | 复制实时流量至测试部署，测量真实场景下的性能 | 高 |

> ⚠️ **推理服务测试成本高昂**。配置测试与测量结果需要工程时间，运行测试流量需要 GPU。
>
> **在一定程度上，这是必要成本，但需仔细考虑如何最小化测试开销**。
>
> **例如**：影子流量测试可先随机复制部分生产流量，再进行短时间负载测试。

> 📌 **测试时需注意**：AI 产品使用量通常有**每日与每周周期波动**。

---

### 5.1 零停机部署

推理工程师使用**高可用部署策略**避免停机。

#### 蓝绿部署（Blue-Green Deployment）

传统高可用设计是**蓝绿部署**。

```
蓝绿部署架构：
  1. 两个完全相同的环境：
     • 蓝色部署（原有）
     • 绿色部署（更新服务）
  2. 绿色环境就绪后，全部流量从蓝色切换至绿色
  3. 蓝色环境保留用于故障回滚
```

> ❌ **但蓝绿部署不适合大规模推理工作负载**，原因与大规模测试相同：**GPU 容量与成本问题**。
>
> 若蓝色部署使用 100 块 GPU，绿色部署在流量切换前需要另外 100 块 GPU。

#### 金丝雀部署（Canary Deployment）

推理工程师可使用**金丝雀部署**，以更低 GPU 开销实现类似效果。

> 💡 灵感来源于煤矿中检测瓦斯的金丝雀，金丝雀部署在影响大量用户前发现错误。

```
┌─────────────────────────────────────────────────────────────┐
│               金丝雀部署流程                                │
├─────────────────────────────────────────────────────────────┤
│  步骤 1：构建推理服务的新部署，准备处理 incoming 请求        │
│         ↓                                                  │
│  步骤 2：将小部分实时流量导向新服务                        │
│         ↓                                                  │
│  步骤 3：监控新服务，确保正确处理流量                      │
│         出现问题立即回滚                                   │
│         ↓                                                  │
│  步骤 4：逐步增加流量，同时监控问题                        │
│         直到新部署处理 100% 流量                           │
└─────────────────────────────────────────────────────────────┘
```

**金丝雀部署的优势**：

| 优势 | 说明 |
|------|------|
| **GPU 开销低** | 不需要双倍 GPU 资源 |
| **灵活推进** | 可快速推进（流量切换仅需几分钟），也可缓慢升级，保证每个阶段稳定 |
| **成本友好** | 结合自动扩缩容，大规模金丝雀部署不会显著增加成本，因为生产系统流量减少会自动缩容部分副本 |

> ⚠️ **启用自动扩缩容后，无流量时新部署默认使用最小副本数**。
>
> 金丝雀部署全过程中，需保证新部署有足够活跃副本处理请求。否则，用户请求会进入队列等待自动扩缩容完成，出现延迟突增。

---

### 5.2 成本估算

从公共 API 按 token 消费，转向专用 GPU 自主推理，需要改变成本思维方式。

#### 公共 API 成本

公共 API 的成本很简单：

```
总成本 = 每百万 token 价格 × 使用 token 数量

有少量变量：
  • 输入 token 的缓存命中 / 未命中
  • 大客户折扣
  • 输出 token 通常比输入 token 更贵

但成本仍是使用量的线性函数。
```

#### 专用推理成本

投入时间与精力做推理工程的动机之一，是掌控单位经济，摆脱按 token 计费模式。但这需要艰难的思维转变。

```
专用推理的优缺点并存：
  ✅ 优势：拥有控制权
  ❌ 劣势：估算难度大

影响成本的因素：
  1. 批处理大小：部署是小批处理优化延迟，还是大批处理优化吞吐量
  2. 流量模式：流量是否持续饱和活跃 GPU，还是存在资源闲置
  3. 序列长度：请求的平均与极端输入、输出 token 数量
```

> 💡 **鉴于这种复杂性，以及输入与输出 token 成本的差异，更高效的方式是将 token 价格转换为总成本，与专用部署成本对比，而非尝试从 GPU 成本反向推算单 token 价格**。

#### 总成本公式

```
┌─────────────────────────────────────────────────────────────┐
│               成本估算公式                                  │
├──────────────────────────────┬──────────────────────────────┤
│  公共 API 总成本             │  专用部署总成本                │
│                              │                              │
│  = 输入 token 数           │  = GPU 成本                  │
│    × 输入单价               │    + 存储成本                │
│    + 输出 token 数         │    + 网络成本                │
│    × 输出单价               │    + 工程时间成本            │
│                              │    + 第三方服务成本          │
│                              │                              │
│  ✅ 简单、线性              │  ⚠️ 复杂、多变量             │
└──────────────────────────────┴──────────────────────────────┘
```

> 📌 **成本估算应使用长周期（理想情况下至少一周），平滑使用量波动**。
>
> **专用部署的另一个成本因素是构建与维护推理系统的工程时间成本**。这项投资虽能提升可靠性、安全性与可控性，但需计入 GPU 成本，形成推理**总拥有成本（TCO）**的完整视图。

---

### 5.3 可观测性

推理是关键业务组件，必须像其他关键业务组件一样监控，内置合适抽象级别的**告警、日志与可观测性**。

#### 首要问题：监控什么？

推理可观测性包括测量：

| 指标 | 说明 |
|------|------|
| **总流量** | 模型部署接收的请求数量 |
| **请求与响应大小** | 处理请求的输入与输出序列长度 |
| **响应码** | 模型服务器返回的 2XX、4XX、5XX 响应码数量 |
| **延迟** | 首 token 时间、token 每秒、P50/P90/P99 端到端延迟等指标 |
| **副本数** | 服务流量的活跃实例数，以及启动中的实例数 |
| **利用率** | CPU、主机内存、GPU、GPU 内存的利用率 |
| **队列深度** | 异步流量系统中，排队等待处理的请求数量 |

> 💡 **这些指标相互关联**。延迟突增可能来自请求流量，也可能来自长输入序列。结合查看这些指标，推理工程师不仅能知道**发生了什么**，还能知道**原因**。

#### 故障排查

故障发生时，推理工程师需要信息解决问题。

```
服务器日志 + 审计日志 = 实时提供故障排查所需信息
  • 服务器日志：记录推理服务的运行状态
  • 审计日志：展示推理服务变更历史
```

#### 集成现有工具

可观测性不能孤立。构建推理可观测性时，需深度集成现有可观测性与告警工具：

| 工具 | 用途 |
|------|------|
| **Grafana** | 指标可视化与仪表盘 |
| **Datadog** | 全栈监控与 APM |
| **PagerDuty** | 告警通知与事件管理 |
| **Sentry** | 错误追踪与性能监控 |

> 💡 将推理信息与应用其他部分关联，才能获得系统整体的可观测性。

## 6. 客户端代码（Client Code）

### 为什么客户端代码很重要？

推理工程涉及众多技术，从 CUDA 到 Kubernetes。但在优化延迟与构建扩缩容能力时，有一个关键领域常被忽视：**客户端代码**。

```
调用推理服务分为两端：

客户端（Client）
  • 向推理引擎发送请求的浏览器、智能体或应用

服务端（Server）
  • 处理客户端请求并返回模型结果的推理服务
```

> 📌 客户端代码的行业标准是 **OpenAI SDK**，除 OpenAI 自有模型外，还支持众多兼容服务商。
>
> LangChain、Vercel AI SDK、LiteLLM、LlamaIndex 等主流 AI 工程框架与库均可作为客户端。

⚠️ **无论使用现有库还是自研代码，都可能存在延迟开销或吞吐量瓶颈**。

---

### 6.1 客户端延迟开销

根据客户端网络连接与使用的协议，客户端与服务端建立会话需要**数十毫秒**。

```
在 P95 端到端延迟 SLA 为 300 毫秒的高性能系统中，
TLS 握手会在推理开始前消耗至少 10% 的延迟预算。
```

> 💡 **解决方案**：同一客户端的后续请求应**复用现有会话**节省时间。

会话复用并非新概念，OpenAI SDK 等工具已在底层静默实现。

> 📌 但为**非标准模态**构建自研客户端时，需遵循会话复用等最佳实践。

---

### 6.2 异步推理

部分系统为**吞吐量而非延迟**构建。

| 场景 | 特点 | 推荐方式 |
|------|------|---------|
| **实时应用** | 延迟敏感 | 同步推理 + 流式传输 |
| **批量文档处理** | 延迟不敏感 | 异步推理 |
| **语料库嵌入** | 延迟不敏感 | 异步推理 |

#### 异步请求机制

异步请求是**"发送即忘"**的推理执行方式。

```
┌─────────────────────────────────────────────────────────────┐
│               同步 vs 异步推理                               │
├──────────────────────────────┬──────────────────────────────┤
│  同步推理                     │  异步推理                       │
│                              │                              │
│  • 客户端发送请求            │  • 立即返回任务 ID             │
│  • 等待响应（有超时时间）     │  • 服务端后台处理              │
│  • 超时后请求失败            │  • 通过 webhook 返回结果      │
│                              │  • 通常有小时级时间限制        │
└──────────────────────────────┴──────────────────────────────┘
```

> 💡 **异步任务仍有时间限制**，但通常以小时为单位，而非分钟。结合稳健的服务端队列，异步请求让高吞吐量、延迟不敏感系统更稳健高效。

---

### 6.3 流式传输与协议支持

流式传输让应用感觉即时响应。

```
对于语言模型，通过 HTTP 流式传输文本输出足够。
但其他模态（尤其是实时语音与视频）的输入与输出流需要传输更多数据。
```

#### 一次性请求 vs 流式传输

| 方式 | 适用场景 | 特点 |
|------|---------|------|
| **一次性 HTTP 请求与响应** | 文本聊天等场景 | 简单，但等待感强 |
| **流式传输** | 实时语音、视频、长文本生成 | 即时响应感，协议更复杂 |

#### 双向流式客户端-服务端连接协议

最常见的双向流式传输协议有两种：

| 协议 | 特点 | 适用场景 |
|------|------|---------|
| **WebSockets** | • 适合无需强 schema 校验的流式场景<br>• 建立持续连接<br>• 服务端支持的客户端数量有开发者配置的上限 | 传输非结构化实时数据（如音频）|
| **gRPC** | • 适合定义完善的服务间通信<br>• 通过 gRPC 传输的请求必须遵循预定义 schema<br>• 无需解析输入<br>• 比 WebSockets 稍慢（额外校验层）| 结构化数据、服务间通信 |

#### WebSockets 详解

```
WebSockets 适用于传输非结构化实时数据（如音频），
接收请求的服务端可解析并下游处理。

使用 WebSockets 时：
  • 服务端支持的客户端数量有开发者配置的上限
  • 达到并发上限后，新连接无法建立
  • 需等待空位释放或新副本扩容
```

#### gRPC 详解

```
与 WebSockets 类似，gRPC 支持双向流式传输，但针对结构化数据。

通过 gRPC 传输的请求必须遵循预定义 schema，无需解析输入。
这一额外校验层让 gRPC 比 WebSockets 稍慢，
但在服务间通信场景中更可靠。
```
