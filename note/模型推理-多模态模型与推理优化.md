## 1. 前言

### 模态定义

模型的模态描述了它接收何种类型的输入、生成何种类型的输出。本系列的第一篇至第五篇重点介绍针对**大语言模型**的推理工程（接收文本输入并生成文本输出）。本文将讨论范围扩展到更多模态。

### 生成式 AI 的模态类型

| 输入 | 输出 | 类别 |
|------|------|------|
| 文本和图像 / 视频 | 文本 | 视觉语言 |
| 文本或图像 / 视频 | 向量 | 嵌入 |
| 音频（语音） | 文本 | 转录（ASR）|
| 文本 | 音频（语音）| 语音合成（TTS）|
| 音频（语音） | 音频（语音）| 语音转语音 |
| 文本和 / 或图像 | 3D 模型 | 生成式计算机辅助设计 |
| 文本和 / 或图像 | 图像 / 视频 | 图像 / 视频生成 |
| 图像 / 视频 | 文本 | 描述 |
| 图像 / 视频 | 掩码 | 分割 |
| 文本和图像 | 图像 | 图像编辑 |

### 两大基本类型

幸运的是，尽管模态众多，但生成式 AI 模型只有两大基本类型：

```
┌─────────────────────────────────────────────────────────────┐
│                生成式 AI 模型两大基本类型                      │
├──────────────────────┬──────────────────────────────────────┤
│  自回归 token 生成    │  迭代去噪                              │
│  • 从 token 化序列开始  │  • 从随机噪声开始                      │
│  • 预测下一个最可能的   │  • 逐步生成最终输出                    │
│    token             │                                      │
└──────────────────────┴──────────────────────────────────────┘
```

**自回归 token 生成模型**：
- 大语言模型是最著名的基于 Transformer 的自回归 token 生成模型
- 视觉语言模型、嵌入模型、ASR 模型、TTS 模型等也依赖类似架构

**迭代去噪模型**：
- 图像和视频生成模型依赖迭代去噪
- 越来越多的混合扩散 Transformer 模型正在刷新质量前沿

> 📌 **关键区别**：虽然从内核选择到参数调优的许多理念也适用于图像模型优化，但具体细节差异很大。

### 衡量指标的差异

对于每一种新模态，你还需要调整对**延迟、吞吐量和质量**的衡量与思考方式。

> **例如**：TTS 模型输出的单个音频 token 并没有实际意义；此时不应使用 TTFT，而应衡量**首个单词生成时间**或**首个句子生成时间**。

## 2. 视觉语言模型（VLM）

### 基本原理

视觉语言模型（VLM）接收一张或多张图像 / 视频以及文本提示，生成文本响应。

```
┌─────────────────────────────────────────────────────────────┐
│                  视觉语言模型组成                             │
├──────────────────────┬──────────────────────────────────────┤
│  大语言模型           │  视觉编码器                           │
│  • 标准大语言模型    │  • 小型模型                           │
│                      │  • 接收原始图像/视频输入              │
│                      │  • 将其转换为图像 token                │
└──────────────────────┴──────────────────────────────────────┘
```

**规模对比**：

| 模型组件 | 典型参数规模 | 示例 |
|---------|-------------|------|
| 视觉编码器 | 较小（~2B）| Mistral Large 3 的视觉编码器 |
| 大语言模型部分 | 较大（~70B+）| Mistral Large 3 的语言模型部分为 73B |

> ⚠️ **尽管视觉编码器的参数规模较小，但对推理至关重要**。视觉语言模型使用多样的视觉编码器架构与实现，因此视觉语言模型的运行时支持较为零散。

### 推理优化的首要挑战：长序列

根据经验，向视觉语言模型输入一张高分辨率图像，会为输入序列增加约 **1000 个视觉 tokens**。

虽然在很高层面上图像 token 与普通 token 类似，但**数量增长极快**。

这为推理的两个阶段都带来了困难：

| 推理阶段 | 挑战 |
|---------|------|
| **预填充（Prefill）** | 图像被分块、嵌入、token 化，并作为输入序列的一部分送入预填充 |
| **解码（Decode）** | 机制相同，但上下文更长；部分模型为图像 token 增加了注意力变体 |

### 适用优化技术

本系列第五篇文章介绍的所有技术都有助于应对这一挑战：

| 技术 | 在 VLM 中的应用 |
|------|-----------------|
| **量化** | KV 缓存量化降低长序列的内存带宽与存储开销 |
| **投机解码** | VLM 的解码与大语言模型一致，可通过投机解码（尤其是 EAGLE）加速 |
| **前缀缓存** | 在多轮对话与重复查询中复用图像对应的 KV 缓存 |
| **并行** | 使用张量并行实现快速推理，同时为大模型与长上下文访问更多显存 |
| **解耦** | 将预填充转移到专用、可独立扩缩容的工作节点，处理长序列 |

### VLM 特有的权衡：降采样

除这些技术外，视觉语言模型引入了新的质量–速度权衡：**降采样**。

| 分辨率 | 所需 token 数 | 信息丰富度 |
|--------|---------------|-----------|
| 高分辨率图像 | 约 4 倍于低分辨率 | 更详细的信息 |
| 低分辨率图像 | 基准 | 信息较少 |

> 💡 **降采样通常对单张图像输入不必要，但在输入多张图像或视频片段时可能需要。**

---

### 2.1 面向视觉语言模型的视频处理

视频不只是帧的集合。视频可能包含音频（尽管许多视觉语言模型无法处理音频，必须单独转录并加入提示词），且帧之间体现物体在空间中的运动，静态图像无法体现。

**关键挑战**：

```
1 秒电影级视频 = 24 帧
每帧高清图像 ≈ 1000 个 tokens
→ 4 秒视频片段 ≈ 96000 个 tokens 的输入序列
```

> ⚠️ 实际上，视频输入不会生成如此长的输入序列——因为**降采样几乎是必须的**。

**视频推理流程**：

```
视频输入
   ↓
降采样（降低分辨率与帧率）
   ↓
token 化与编码
   ↓
推理（与图像处理类似，但上下文更长）
```

> 📌 **重要**：视频被 token 化与编码后，推理与图像处理类似，只是上下文更长。前缀缓存、KV 缓存卸载与优化注意力实现对这些数万 tokens 的输入序列更为重要。

---

### 2.2 全模态模型（Omni-Modal Models）

视觉语言模型是"全模态"模型趋势的重要组成部分，这类模型接收多种类型输入并生成多种类型输出。

**全模态模型的利弊**：

| 优点 | 缺点 |
|------|------|
| 混合模态提供独特能力 | 更小的专用模型通常在特定领域更快、更准确 |
| 统一架构简化系统 | 质量可能不如专用模型 |

> **例如**：许多视觉语言模型的图像输入处理中内置了文本识别能力。但这些能力仍落后于**专用光学字符识别（OCR）模型**，而后者的规模通常仅为前者的几分之一。

**生产环境推荐架构**：

```
生产环境中运行视觉语言模型推理，
通常需要协同多个模型与预处理步骤的流水线：

PDF 数据提取 → OCR 文本读取 → 视觉语言模型理解
    ↓
视频音频转录 → 加入提示词 → 视觉语言模型理解
```

> 💡 流水线中的每个组件都必须**单独进行速度优化**，并**独立扩缩容**，以避免瓶颈。

## 3. 嵌入模型（Embedding Models）

### 基本原理

嵌入模型将变长文本块（或图像等其他模态输入）转换为**固定长度向量**表示，以捕捉输入的语义含义。

```
┌─────────────────────────────────────────────────────────────┐
│                  嵌入模型工作原理                           │
├─────────────────────────────────────────────────────────────┤
│  输入：变长文本 / 图像                                     │
│         ↓                                                  │
│   嵌入模型（基于 Transformer）                               │
│         ↓                                                  │
│   输出：固定长度向量（如 768 维、1536 维）                 │
│         ↓                                                  │
│   应用：向量相似度计算（余弦相似度、欧氏距离等）            │
└─────────────────────────────────────────────────────────────┘
```

通过将内容编码到共享语义向量空间，你可以用简单数学比较项目之间的距离。

**主要应用场景**：
- Agent Memory（智能体记忆）
- RAG（检索增强生成）
- 搜索系统
- 推荐系统

---

### 3.1 嵌入模型架构

Hugging Face 上有数万种嵌入模型，但都使用两种基于 Transformer 的架构之一：

| 架构类型 | 说明 | 典型规模 | 适用场景 |
|---------|------|---------|---------|
| **BERT 类模型** | 仅编码器神经网络，最初为掩码 token 预测构建 | < 1B 参数 | 分类等对延迟敏感的简单任务 |
| **基于大语言模型的模型** | 现代语言模型，被改造用于生成嵌入 | ≤ 8B 参数 | 需要强语义理解能力的场景 |

> 📊 **现状**：基于大语言模型的嵌入模型提供显著更强的能力，尽管 BERT 类模型仍用于分类等对延迟敏感的简单任务。

#### 嵌入维度：速度–质量权衡

嵌入模型在**嵌入维度**（输出向量长度）上引入自身的速度–质量权衡：

| 向量维度 | 信息编码能力 | 存储开销 | 检索计算时间 |
|---------|-------------|---------|-------------|
| 短向量（如 384 维）| 较低 | 小 | 快 |
| 长向量（如 1536 维）| 较高 | 大 | 慢 |

**Matryoshka 表示**：

现代嵌入模型使用 **Matryoshka 表示**，在维度与质量之间实现动态权衡，同时在更短向量上保留更多信息。

> 💡 **重要**：维度基本不影响推理时间，但会影响系统中的存储、检索与相似度计算时间。

#### 向量空间兼容性

> ⚠️ **注意**：在大多数情况下，一个嵌入模型的向量无法与另一个嵌入模型的向量进行有意义比较，即使长度相同，因为它们将输入编码到**不同语义空间**。

---

### 3.2 嵌入模型推理

嵌入模型的推理工作负载有两种不同的流量模式：

| 流量模式 | 说明 | 特点 |
|---------|------|------|
| **高吞吐量回填** | 批量操作，如索引数百万份文档、更新产品目录、为大语言模型预训练准备数据 | 延迟不敏感，吞吐量优先 |
| **低延迟查询** | 面向用户的独立搜索、检索或推荐请求 | 每毫秒延迟都会影响用户体验 |

> 💡 **建议**：如果你需要同时支持两种模式且流量足够大，值得为每种使用场景构建**独立系统**。

#### 推理运行时选择

| 运行时 | 特点 | 推荐度 |
|--------|------|--------|
| **TensorRT-LLM** | 优化的 XQA 内核实现快速注意力，内核融合技术减少内存访问开销 | ⭐⭐⭐⭐⭐ 最佳性能 |
| **vLLM** | 支持嵌入模型，生态完善 | ⭐⭐⭐⭐ 推荐 |
| **SGLang** | 支持嵌入模型，性能优秀 | ⭐⭐⭐⭐ 推荐 |
| **Infinity** | 专用嵌入推理运行时 | ⭐⭐⭐ 可选 |
| **TEI**（Hugging Face 文本嵌入推理）| Hugging Face 官方嵌入推理引擎 | ⭐⭐⭐ 可选 |

**TensorRT-LLM 的优势**：

```
TensorRT-LLM 提供：
  1. 优化的 XQA 内核 → 快速注意力计算
  2. 内核融合技术 → 减少内存访问开销
  3. 最新架构支持（Hopper、Blackwell）→ 充分发挥硬件性能

对于受支持的模型，TensorRT-LLM 在延迟与吞吐量上
都是性能最强的推理引擎。
```

#### 量化

进一步提升来自量化：

```
虽然小模型更容易因量化损失质量，
但对嵌入模型权重进行 FP8 量化可在最小质量损失下提升性能。
```

**质量检查方法**：

> 检查量化后嵌入模型质量的最简单方法是：将相同输入同时送入原始模型与量化模型，然后检查输出向量的**余弦相似度**。
>
> - 余弦相似度为 100% = 向量完全相同
> - 你希望相似度至少达到 **99%**，以确保量化可靠

#### 不适用于嵌入模型的优化

| 优化技术 | 是否适用 | 原因 |
|---------|---------|------|
| **前缀缓存** | ❌ 不适用 | 嵌入模型并行处理 token，无自回归生成 |
| **解耦** | ❌ 不适用 | 嵌入模型无预填充/解码分离需求 |
| **跨多 GPU 并行** | ❌ 效率不高 | 这些模型规模较小 |
| **水平扩展（副本）** | ✅ 推荐 | 高流量部署应水平扩展，每块 GPU 作为独立副本 |

#### 批处理与排队

由于嵌入模型并行处理 token，批处理与排队对性能起重要作用：

```
嵌入模型提供的批处理大小远高于其他模型：
  - 单个请求可将数十到数百个文本输入批量组合为列表
  - 许多请求可在单块 GPU 上并行运行
  - 因为即使最复杂的嵌入模型也相对小巧、快速
```

> ⚠️ **流量管理**：无论你执行大规模回填还是应对使用量激增，流量都可能超过嵌入模型提供的大批处理大小。在这些情况下，**稳健的排队系统**是支持嵌入模型推理的必要基础设施。

## 4. 自动语音识别模型（ASR）

### 基本原理

自动语音识别（ASR）模型接收音频输入并生成文本输出，支撑转录与听写应用。

```
┌─────────────────────────────────────────────────────────────┐
│                  ASR 模型：Whisper 架构                    │
├──────────────────────┬──────────────────────────────────────┤
│  编码器              │  解码器                               │
│  • 接收处理后的     │  • 接收编码音频特征                   │
│    音频波形          │  • 将其转换为文本 token                │
│  （对数梅尔频谱图） │  • 自回归 Transformer 模型            │
└──────────────────────┴──────────────────────────────────────┘
```

**Whisper 模型特点**：

| 特点 | 说明 |
|------|------|
| 开源 | OpenAI 发布，支持数十种语言的准确转录 |
| 尺寸多样 | 从 Tiny 到 Large，最大模型仅 1.55B 参数 |
| 运行环境 | 可以在 H100 等大型 GPU 的分片（通过多实例 GPUs / MIGs）上极快地运行 |
| 推荐版本 | Whisper 3 Large 与 Whisper 3 Turbo 可满足大多数延迟预算 |

> 📌 **关键**：绝大多数推理时间花费在**解码器**上。解码器是自回归 Transformer 模型，架构与大语言模型非常相似。幸运的是，有优秀工具可以优化这一主要瓶颈。

### 解码器端性能优化

解码器端性能优化的主要工具是 **TensorRT-LLM**：

```python
# 使用 TensorRT-LLM 优化 Whisper 解码器
from tensorrt_llm import ModelRunner

runner = ModelRunner.from_dir(
    engine_dir="./whisper-trt-engin",
    lora_dir=None
)
# 动态批处理 + 优化 CUDA 内核
```

**TensorRT-LLM 的优势**：
- 为解码器实现**动态批处理**
- 配备高效 CUDA 内核的优化 C++ 运行时
- 与 Hopper、Blackwell 等最新架构配合效果尤佳
- 使 MIG（多实例 GPU）成为 ASR 推理的更佳选择

---

### 4.1 单片段延迟优化

#### 实时转录

Whisper 的一个用例是**实时转录**，如听写应用或语音助手。

**关键指标**：

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **单片段延迟** | 200 毫秒 | 人类平均反应时间 |
| **实时因子（RTF）** | < 1 | 处理时间 / 音频时长 |

> 💡 **在优化的 TensorRT-LLM 推理引擎上运行 Whisper 时，运行时层面提升性能的空间已经不大**。相反，实时语音识别的大部分提升来自**编排与基础设施**。

#### 流式传输架构

ASR 产品体验的最大升级是**流式传输**，它在 API 服务器层实现，而非模型运行时层。

```
流式转录实现：

音频流 → WebSocket 连接 → VAD 语音活动检测
                              ↓
                         分割为独立片段
                              ↓
                         ASR 模型处理（常规方式）
                              ↓
                         文本结果通过 WebSocket 流式返回
```

> ✅ **优势**：
> - 产品可以实时转录，而非转录预录制音频
> - 保持转录顺序
> - 当每个片段在同一块 GPU 上处理时，你可以将前一片段的输出序列作为下一片段的前缀，**提升转录质量**

---

### 4.2 长文件延迟优化

#### Whisper 的限制

Whisper 模型的一个限制是只能支持 **30 秒片段**。

转录长文件（如一小时播客）需要一套不同的优化方案。

#### 实时因子（RTF）

使用**实时因子（RTF）**衡量长文件转录性能：

```
RTF = 处理时间 / 音频时长

示例：
  - 世界上最快的打字员需要 30 分钟手动转录 1 小时音频
    → RTF = 30/60 = 0.5 倍（即 2 倍速）
  - 经过长文件优化的 Whisper 部署可以在不到 4 秒内转录 1 小时音频
    → RTF = 4/3600 ≈ 0.0011
    → 约 900 倍加速
```

#### 长文件转录流水线

```
┌─────────────────────────────────────────────────────────────┐
│            长音频文件转录的两阶段流水线                       │
├─────────────────────────────────────────────────────────────┤
│  阶段 1：VAD 模型                                         │
│  • 运行在专用硬件上                                        │
│  • 去除静音，并将音频切分为有意义的片段                     │
│  • 按语义分割（避免将单词切成两半）                        │
│                         ↓                                  │
│  阶段 2：并行片段转录                                      │
│  • 片段可以并行处理                                        │
│  • 使用多块 GPU（或多个 MIG）同时处理更多音频片段           │
│  • RTF 大致随使用的 GPU 数量线性提升                       │
│  • 每块 GPU 通过动态批处理同时处理多个片段                  │
│                         ↓                                  │
│  阶段 3：结果拼接                                          │
│  • 按时间戳将分块转录结果拼接回去                          │
└─────────────────────────────────────────────────────────────┘
```

> ⚠️ **并行片段转录的代价**：会失去使用前序序列作为后序前缀的能力。但有其他质量提升技术可以弥补。

#### 质量提升技术

对于 ASR 输出，你可以通过测量输出的压缩比与每分钟单词数，自动检测重复单词、短语等幻觉。

当片段出现问题时，你可以：

| 方法 | 说明 |
|------|------|
| **以更高温度重新运行片段** | 看似违反直觉（更高温度通常产生更多幻觉），但目的是打破重复循环，生成不同输出 |
| **重新切分音频片段** | 将整个音频或音频片段重新切分为更小片段，重新转录 |

> ✅ 在实践中，这些技术消除了使用前序序列作为前缀的需求，实现高效、准确的长文件并行转录。

---

### 4.3 说话人分离

**说话人分离**（为转录标注说话人与时间）是与转录相关的问题。

#### 说话人分离模型

说话人分离模型根据声音特征对音频进行分类，然后在文件中分段与聚类，标注说话人变化时间。

| 模型类型 | 说明 |
|---------|------|
| **Whisper**（ASR）| 编码器–解码器 Transformer 模型 |
| **pyannote.audio** | 说话人分离系统，经典机器学习模型流水线 |

#### 说话人分离流水线

说话人分离流水线包含分割、嵌入与聚类模型。

```
说话人分离流水线：

音频输入
   ↓
分割模型（Segmentation Model）
   ↓
嵌入模型（Embedding Model）→ 提取声音特征向量
   ↓
聚类算法（Clustering）→ 标注说话人变化时间
```

> ⚠️ **性能特点**：由于说话人分离是机器学习流水线，你可以使用 PyTorch、pyannote 等工具以及 Torch 编译等优化方法提升性能。但即使高度优化的说话人分离实现，处理音频文件的时间也**至少是转录的两倍**。

## 5. 文本转语音模型（TTS）

### 基本原理

文本转语音（TTS）模型（又称语音合成模型）接收文本输入并生成音频输出，专门生成语音。

```
┌─────────────────────────────────────────────────────────────┐
│                  TTS 模型：Orpheus TTS                     │
├─────────────────────────────────────────────────────────────┤
│  文本输入                                                  │
│      ↓                                                     │
│  TTS 模型（基于 Llama 3.2 3B 微调）                       │
│  • 词表大小扩展数万，编码音频令牌                         │
│      ↓                                                     │
│  音频令牌输出                                              │
│      ↓                                                     │
│  音频解码器（SNAC Decoder）                                │
│      ↓                                                     │
│  音频波形输出                                              │
└─────────────────────────────────────────────────────────────┘
```

**现代 TTS 模型特点**：

| 特点 | 说明 |
|------|------|
| 基于大语言模型 | 现代 TTS 模型是微调大语言模型（如 Orpheus TTS 源自 Llama 3.2 3B）|
| 参数规模较小 | 3B 参数的 Orpheus TTS 属于较大型号 |
| 推理运行时 | H100 上的 MIG 是高效、高性能的推理选择 |
| 量化支持 | TTS 模型权重与 KV 缓存可以量化为 FP8 |

> 📌 **重要**：TTS 模型通过将大语言模型词表大小扩展数万（编码音频令牌）进行训练。随后，模型在文本输入与令牌化音频输出配对数据上训练。这意味着在实际使用 TTS 模型时，你还需要**音频解码器**，将音频输出 token 转换为波形。

---

### TTS 模型推理优化

由于 TTS 模型源自大语言模型，许多为大语言模型开发的运行时与性能优化技术同样适用：

| 优化技术 | 说明 |
|---------|------|
| **TensorRT-LLM** | 优化推理引擎，提供动态批处理与高效 CUDA 内核 |
| **量化（FP8）** | TTS 模型权重与 KV 缓存可以量化为 FP8，进一步提升性能 |
| **Torch 编译** | 编译音频解码器（SNAC Decoder），针对目标 GPU 优化 |

> ⚠️ **音频解码器的瓶颈**：这一音频解码过程为推理增加了潜在瓶颈。音频解码器应使用 PyTorch 实现，并针对目标 GPU 编译高效运行，同时使用动态批处理与短超时（如 15 毫秒）。

---

### TTS 性能指标

TTS 模型性能使用与大语言模型略有不同的指标衡量：

| 指标 | 全称 | 说明 |
|------|------|------|
| **TTFB** | Time To First Byte | 首字节时间，是语音合成的首 token 时间等效指标 |
| **首句时间** | Time To First Sentence | 比 TTFB 更贴近用户体验的延迟指标，即生成第一个有意义短语或句子的时间 |
| **TPS** | Tokens Per Second | TTS 模型生成 token 的速度，与大语言模型一样可以用每秒 token 数衡量 |

#### TTFB 目标

与大语言模型的 TTFT 一样，语音合成的目标是**最小化 TTFB**。

> 对于 Orpheus，在单块 H100 上可以低至 **150 毫秒**。

#### TPS 目标

但是，TTS 模型的 TPS 目标有所不同。

```
模型生成的 token 会转换为音频波形。
根据模型不同，实时生成音频可能需要每秒 80 至 100 个 token。
超过该水平，每秒生成更多 token 没有额外收益。
```

> 💡 **性能提升方向**：相反，性能提升用于**扩展并发实时输出数量**。如果单块 GPU 可以支持更多并发用户，语音合成的单用户成本会大幅下降。

---

### 5.1 流式实时文本转语音

大多数文本转语音任务需要**实时语音合成**。

> 📌 **与 ASR 模型一样，实时系统的性能提升更少来自运行时层（已通过 TensorRT-LLM、量化与编译 SNAC 解码器优化），而更多来自基础设施**。

**流式传输架构**：

```
基于 WebSocket 的流式传输：

客户端 ─┐
        │ WebSocket 连接
服务器 ─┘
        ↓
   TTS 模型生成音频 token
        ↓
  音频解码器转换为波形
        ↓
  通过 WebSocket 流式返回音频
```

> 💡 **部署建议**：测试推理引擎确定可生成的并发实时流数量后，设置相同的批处理大小与活跃 WebSocket 数量，保持高使用率但稳定。

**批量用例**：

TTS 模型很少在实时应用之外使用。但是，如果你确实遇到批量用例（如为大量文档回填音频以提升可访问性），请注意 **TTS 模型不适合处理长输入**，语音质量在 30 秒左右开始下降。

---

### 5.2 语音转语音模型

#### 级联方式（当前主流）

如今，大多数语音系统使用**级联方式**：

```
语音输入
   ↓
ASR 模型（语音 → 文本）
   ↓
大语言模型（文本理解 + 文本生成）
   ↓
TTS 模型（文本 → 语音）
   ↓
语音输出
```

这些流水线还使用 VAD、嵌入模型等辅助组件，促进自然对话并添加上下文。

#### 语音转语音模型（新兴方向）

OpenAI 的 **gpt-realtime** 等语音转语音模型，为核心大语言模型增加音频接收与生成能力，有效将流水线统一为单一模型。

```
语音转语音模型：

语音输入
   ↓
统一模型（接收音频 + 生成音频）
   ↓
语音输出
```

> 📊 **现状（截至 2026 年初）**：
> - 尚无商业可用的开源语音转语音模型
> - gpt-realtime 等闭源方案能力明显弱于级联多模型方案，且成本更高
> - 但该领域研究活跃，这一新兴模态很快将需要专属推理工程方案

## 6. 图像生成模型

### 基本原理

在几个方面，处理图像与视频生成模型与处理大语言模型完全不同。

| 对比维度 | 大语言模型 | 图像 / 视频生成模型 |
|---------|------------|-------------------|
| **架构** | 自回归 token 生成器（Transformer）| 迭代去噪器（Diffusion）|
| **工具链** | 成熟（vLLM、TensorRT-LLM 等）| 较新（SGLang Diffusion、vLLM Omni）|
| **约束条件** | 受内存带宽限制 | 受计算限制 |
| **质量评估** | 相对容易（困惑度、基准测试）| 非常困难（需要人工评估）|

> 📌 **关键区别**：图像与视频生成模型是多个小型模型在隐空间协同工作的流水线，而非大语言模型的统一解码器架构。

### 质量–速度权衡

也许最重要的区别是，图像与视频生成模型提供更直接的**质量–速度权衡**。

```
以编程方式评估图像模型输出质量非常困难：
  - 使用视觉语言模型的自动流水线最多只能给出方向性信号
  - 可能与人类偏好不一致
  - 人眼的评判难以捉摸
  - 大多数图像质量评估通过让人类在数千张图像中选择，
    将感受与偏好汇总为质量基准
```

---

### 6.1 图像生成内核优化

当你从模型仓库阅读图像生成模型的模型卡片时，推理示例通常使用 **diffusers 库**，几乎没有优化。

> 💡 **实际上，尽管理论上图像生成是计算密集型，但你通常需要选择内存高效内核并使用内核融合，才能真正达到这一瓶颈**。

#### 高性能图像模型推理库

| 库 | 特点 | 推荐度 |
|-----|------|--------|
| **SGLang Diffusion** | 为流行图像与视频生成架构构建的高性能推理引擎 | ⭐⭐⭐⭐⭐ 推荐 |
| **TensorRT** | 高质量黑盒实现，使用 NVIDIA 自研内核运行主流模型 | ⭐⭐⭐⭐ 推荐 |
| **PyTorch** | 精心选择内核与融合，提供可控、灵活、高端性能提升 | ⭐⭐⭐⭐ 适合深度定制 |

> 💡 **建议**：
> - 如果你希望立即获得可用效果，直接使用 **SGLang Diffusion** 或 **TensorRT** 实现即可
> - 使用 **PyTorch**，高级推理工程师有机会进行深度定制

#### 关键内核优化

| 内核类型 | 说明 | 推荐实现 |
|---------|------|---------|
| **注意力内核** | 最核心的内核 | FlashAttention 2/3/4（Hopper/Blackwell GPU）|
| **归一化函数** | RMSNorm 等，融合的良好候选 | 手动融合确保内存使用高效 |
| **GEMM 内核** | 应用于线性层，通常可以安全量化为 8 位浮点格式 | CuTe、CUTLASS 或 DeepGEMM |

#### Torch 编译

Torch 编译包含自动内核融合与插件系统，可插入手动选择的内核。

```python
import torch

# 编译模型以获得优化性能
compiled_model = torch.compile(
    model,
    mode="max-autotune",  # 最大优化
    fullgraph=True         # 完整图编译
)
```

> ⚠️ **重要**：
> - 生成的引擎可以缓存，加快节点启动加载速度（这一点很重要，因为编译需要数分钟）
> - Torch 编译针对执行编译的特定 GPU 型号与架构——**如果你希望在 B200 上运行模型，请在 B200 上完成编译**

---

### 6.2 加速图像生成的一个实用技巧

内核选择与 Torch 编译都是真正的推理优化技术。但推理优化世界中也有有趣的**技巧**。

#### 图像生成是分步过程

```
图像生成时间与步数线性相关：

50 步模型 → 生成时间基准
25 步模型 → 生成时间减半
  但质量可能降至不可接受水平
```

这就是为什么**少步模型**与**隐一致性模型**比完整 50 步模型快得多的原因。

#### 无分类器引导（Classifier-Free Guidance）

每一步通过去噪模型的批处理大小为 2，因为每一步包含：
- 一次**带提示引导**的迭代
- 一次**不带提示引导**的迭代

**引导参数**控制在合并每一步生成的两次迭代时，带提示引导图像的权重。

> 如果引导为 0，则无需生成带提示引导的图像（节省一半计算）。

#### 实用技巧：中途关闭引导

回顾一下，图像的大致轮廓在**早期步骤**确立，剩余步骤用于填充细节。

```
因此，提示相关性在早期步骤更重要：
  - 这些步骤影响图像整体轮廓
  - 模型不会在后续步骤中改变主意（不会把正在生成的猫换成狗）
```

**优化方法**：

```
如果你在图像生成中途关闭引导：
  - 可以在不减少步数的情况下减少去噪模型的运行次数
  - 如果在 50 步运行的最后 20 步跳过引导
  - 模型只需运行 80 次而非 100 次
  - 质量通常保持高位
```

## 7. 视频生成模型

### 基本原理

视频生成是要求最高的模态。只要条件允许，这些模型应在 **Blackwell GPU**（或未来可用的 Rubin）上运行。

> 💡 **Blackwell GPU 的优势**：
> - 高内存容量用于上下文并行
> - 快速张量核心用于注意力计算
> - 微缩放数据格式（MXFP8、MXFP4）以实现更精确量化

### 与图像生成的异同

| 维度 | 图像生成 | 视频生成 |
|------|---------|---------|
| **架构** | 迭代去噪，隐空间表示二维（宽、高）| 迭代去噪，隐空间表示三维（宽、高、时间）|
| **计算量** | 中等 | 大得多（每一步处理的数据量大得多）|
| **注意力计算占比** | 中等 | 70%~80% 的计算时间 |
| **并行策略** | 张量并行 | 上下文并行 |

> 📌 **关键**：视频生成模型通常采用与图像生成模型相同的去噪步数（约 50 步），但每一步处理的数据量大得多。

### 视频生成的资源需求

由于视频生成模型是计算密集型的，**批处理不像文本生成那样有用**。

```
视频生成模型通常在整台 8 卡节点上运行，批处理大小为 1：
  - 全部 8 块 GPU 协同生成一个视频
  - 而非像 LLM 推理那样每块 GPU 处理不同请求
```

> ⚠️ **与可以通过调整批处理大小实现延迟–吞吐量权衡的批量工作负载不同，提升视频生成吞吐量与降低成本的唯一方法是让模型本身更快**。

### 视频质量演进

早期视频生成模型是**逐帧的**。它们一帧一帧生成。这降低了视频输出的质量与连贯性。

如今，视频生成模型在**隐空间中对整个视频**运行去噪步骤。

---

### 7.1 注意力优化与量化

注意力优化从**内核选择**开始。

**推荐内核**：

| 内核 | 适用场景 |
|------|---------|
| **FlashAttention** | 通用注意力优化 |
| **DeepGEMM** | 高效 GEMM 计算 |
| **CuTe** | 灵活内核定制 |
| **CUTLASS** | NVIDIA 官方高性能模板库 |

> 📌 **测试建议**：测试 FlashAttention、DeepGEMM、CuTe、CUTLASS 内核，查看哪种最适合你的模型。

#### 缓存方法

语言模型使用 KV 缓存加速注意力，视频生成模型则使用**其他缓存模式**尝试复用模型输出。

复用注意力计算的部分内容可以使视频生成速度实际提升 **30%~40%**。

**两种基本缓存方法**：

| 缓存方法 | 说明 | 质量损失 |
|---------|------|---------|
| **基于时间步的缓存** | 缓存并复用特定时间步的输出，跳过整个步骤 | 从可忽略到中等 |
| **基于 Transformer 的缓存** | 缓存并复用隐状态，跳过 Transformer 自身的层 | 从中等到高 |

> ⚠️ **生产使用前请仔细测试**。这些策略的质量损失从可忽略到无法使用不等。

#### 量化策略

除内核与缓存外，加速注意力的主要工具是**量化**。

| 对比维度 | 语言模型量化 | 视频生成模型量化 |
|---------|-------------|-----------------|
| **量化重点** | 权重（大型线性层）| 注意力 |
| **量化收益来源** | 需要加载到内存的数据更少（带宽优化）| 切换到低精度张量核心，获得双倍 FLOPS（计算优化）|
| **最敏感组件** | 注意力层 | 注意力层 |
| **风险** | 误差在数千次自回归迭代中累积 | 只有约 50 步，误差累积风险略低 |

**推荐的量化方法**：

| 方法 | 说明 |
|------|------|
| **分块量化** | 与微缩放数据格式（MXFP8）配合，在 Hopper 与 Blackwell 上支持 |
| **按步骤量化** | 早期步骤保持 FP16，后续步骤量化（遵循与图像生成中无分类器引导技巧相同的思路）|
| **按层量化** | 保留首层与末层，量化隐藏层 |
| **SageAttention** | 8 位注意力内核，可用于视频生成模型的高质量低精度注意力 |

> 💡 **按步骤量化原理**：早期步骤确立图像轮廓，后续步骤优化细节。这些早期步骤对提示相关性与准确性更重要。

> 💡 **按层量化原理**：首层与末层更重要，因为它们接收输入并生成最终输出。隐藏层仅执行中间计算，受近似影响较小。

---

### 7.2 上下文并行

尽管视频生成模型通常在整台 8 卡节点上运行，它们使用**上下文并行**而非张量并行。

```
┌─────────────────────────────────────────────────────────────┐
│                  上下文并行工作原理                           │
├─────────────────────────────────────────────────────────────┤
│  张量并行：在 GPU 间拆分模型权重                             │
│  上下文并行：复制权重到每块 GPU，但在 GPU 间拆分注意力计算     │
│                                                           │
│  实现机制：环形注意力                                       │
│  • 每块 GPU 持有一部分上下文                                │
│  • 将中间结果传递给环中的下一块 GPU                         │
└─────────────────────────────────────────────────────────────┘
```

> 💡 **为什么使用上下文并行而非张量并行？**
> 视频模型足够小，将权重复制 8 份会占用可观内存，但在 B200 上可行。上下文并行避免了拆分权重带来的通信开销。

#### 可并行的内容

注意力不是唯一可以并行的部分。

| 可并行组件 | 占总推理时间比例 | 说明 |
|-----------|----------------|------|
| **注意力** | 70%~80% | 多头注意力，头相互独立，可分开运行后合并结果 |
| **隐解码步骤**（使用变分自编码器）| 3%~5% | 可以跨 GPU 运行 |

> 📈 **这些并行技术使 AI 视频成为可能。随着视频序列变长、视频模型变大，并行将继续成为视频生成模型推理最关键的技术**。

## 总结

本文介绍了除大语言模型外的六种常见模态的推理工程：

```
┌─────────────────────────────────────────────────────────────┐
│                  多模态推理技术总结                           │
├──────────┬──────────────────────────────────────────────────┤
│  视觉语言 │  长序列挑战，适用量化/投机解码/前缀缓存/并行/解耦  │
│  嵌入模型 │  高吞吐量与低延迟两种模式，批处理是关键            │
│  ASR     │  实时流式传输，长文件并行转录                      │
│  TTS     │  实时流式合成，音频解码器是瓶颈                    │
│  图像生成 │  迭代去噪，内核优化与技巧（中途关闭引导）          │
│  视频生成 │  计算密集型，注意力优化+量化+上下文并行            │
└──────────┴──────────────────────────────────────────────────┘
```

### 跨模态通用优化技术

| 技术 | 适用模态 |
|------|---------|
| **量化（FP8）** | 全部（VLM、嵌入、ASR、TTS、图像、视频）|
| **TensorRT-LLM** | VLM、嵌入、ASR、TTS |
| **流式传输架构** | ASR、TTS |
| **并行技术** | VLM（张量并行）、视频（上下文并行）|
| **批处理优化** | 嵌入模型（高批处理大小）|
