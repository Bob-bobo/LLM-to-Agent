## 1. 前言

推理工程最令人兴奋的一点是：与许多学术研究成果需要数年甚至数十年才能被工业界采用不同，新论文中的技术在短短几周甚至几个月内就能应用于生产环境。

> **推理工程核心原则**：在推理系统中引入的约束条件越多，就能获得越好的性能。

例如解耦技术，它允许将单个引擎的计算约束到预填充（Prefill）和解码（Decode）两个阶段分别优化。

### 使用性能技术的两个重要原则

**原则一：流量越大，优化空间越大**

你的流量越大，能做的性能优化就越多（同时保持合理的单位经济性）。例如：
- 在更多 GPU 上实现更高的模型并行度
- KV 感知路由
- 动态解耦

这些优化只有在你拥有大量 GPU（通常是多个节点）、以垂直扩展和水平副本方式为同一个模型提供服务时才有意义。

**原则二：技术之间可能相辅相成，也可能互不兼容**

- ✅ 量化 KV 缓存 → 缓解解耦的瓶颈
- ❌ 增大批处理大小 → 减少可用于推测解码的计算量

推理工程师的目标始终是创建一组**平衡的优化方案**，使其整体效果大于各部分之和。

### 本文覆盖的五类关键技术

```
量化 → 投机解码 → 缓存 → 并行 → 解耦
（计算优化）  （解码加速）  （内存优化）（扩展推理）（架构分离）
```

## 2. 量化（Quantization）

量化可以提升延迟（包括 TTFT 和 TPS）、提高系统吞吐量，并为解耦、投机解码、前缀缓存等其他优化提供更大空间。

> ⚠️ **注意**：一旦出错，量化会显著降低模型的输出质量。

### 2.1 数值格式

#### 量化工作原理

训练后量化（PTQ）通过将模型权重和其他数值从原生格式转换为更低精度的格式来工作。

**原生训练精度**：通常是 BF16 或 FP16，8 位和 4 位原生精度在训练中也越来越流行。

**量化带来的性能提升**：

| 推理阶段 | 优化效果 |
|---------|---------|
| **预填充（Prefill）** | 计算密集型的预填充运行在低精度张量核心上，FLOPS 翻倍 |
| **解码（Decode）** | 内存密集型的解码每个数值加载的数据量减半，内存带宽实质翻倍 |

> 💡 **实际效果**：从 16 位降到 8 位并不会线性地快 2 倍，单级精度降低通常能为 LLM 带来 **30%~50%** 的性能提升。

#### 常见数值格式对比

| 名称 | 缩写 | NVIDIA GPU 首次支持架构 | 年份 |
|------|------|------------------------|------|
| 64 位浮点 | FP64 | Fermi | 2010 |
| 32 位浮点 | FP32 | Kepler | 2012 |
| 16 位浮点 | FP16 | Pascal | 2016 |
| 脑浮点 16 位 | BF16 | Ampere | 2020 |
| 8 位浮点 | FP8 | Hopper | 2022 |
| 混合精度 FP8 | MXFP8 | Blackwell | 2024 |
| 8 位整数 | INT8 | Pascal | 2016 |
| 6 位浮点 | FP6 | Blackwell | 2024（实验性）|
| 4 位浮点 | FP4 | Blackwell | 2024 |
| 混合精度 FP4 | MXFP4 | Blackwell | 2024 |
| NVIDIA FP4 | NVFP4 | Blackwell | 2024（专有）|
| 4 位整数 | INT4 | Turing | 2018 |

> **推理主力格式**：16 位、8 位、4 位精度是推理的主要格式。FP64 仅用于高精度科学计算，不用于 AI 推理。

#### 浮点 vs 整数格式

浮点格式比整数格式更适合推理，原因是：

```
浮点数 = 符号位 + 指数位 + 尾数位
整数格式 = 符号位 + 数值位
```

**浮点数的优势**：指数使其拥有更高的动态范围，能更好地表示极大和极小的数值。这一点很重要，因为**异常值在推理中意义重大**，而浮点格式能在量化后更好地保留异常值。

以 E4M3 格式的 FP8 为例：
- 4 位指数
- 3 位尾数
- 1 位符号位

#### 量化粒度级别

量化可以应用在不同级别，粒度越细，质量保留越好，但开销越大：

| 粒度级别 | 说明 | 质量 | 开销 |
|---------|------|------|------|
| **张量级别** | 为整个 QKV 张量计算单个缩放因子 | 低 | 低 |
| **通道级别** | 为张量内每个特征向量计算不同缩放因子 | 中 | 中 |
| **分块级别** | 在每个特征向量内，将向量切分为 N 个数值的块，为每个块计算缩放因子 | 高 | 高 |

**Blackwell 架构的新格式**：
- **MXFP8 / MXFP4**：微缩放格式，每 32 个参数计算一个分块缩放因子
- **NVFP4**：NVIDIA 专有格式，块大小为 16，并附带 32 位全局缩放因子

> ⚠️ 微缩放格式的代价：小块缩放因子也必须存在内存中，略微降低量化带来的性能收益。

#### 本地/边缘推理的特殊情况

- **GGUF**：存储模型的二进制格式，是 Hugging Face 上分发高度量化模型的最流行选择
- 研究者能把 DeepSeek 这类巨型模型压缩到苹果电脑等消费硬件上运行
- 这些量化策略通过**动态量化**对抗质量下降：模型的某些层或组件保留原始精度，其他部分则被量化到低至 1 比特精度的整数

#### 生产环境推荐

| 场景 | 推荐格式 |
|------|---------|
| **生产环境（质量敏感）** | FP8、MXFP8（浮点格式） |
| **本地/边缘推理** | 动态量化（如 1.58 比特量化） |
| **不推荐** | INT 格式（动态范围不足，不适合质量敏感工作负载） |

> 💡 **FP8 和 MXFP8 提供最大的灵活性，尤其在量化 KV 缓存时。**

### 2.2 量化方法

#### 量化时机选择

| 方法 | 全称 | 说明 |
|------|------|------|
| **QAT** | Quantization-Aware Training（量化感知训练）| 同时训练权重和计算缩放因子，确保最终收敛的权重在指定精度下准确 |
| **PTQ** | Post-Training Quantization（训练后量化）| 通过计算缩放因子，将训练完成的模型权重转换为新精度，并通过校准保留精度 |

> 📌 使用开源模型的推理工程师通常只能进行**训练后量化（PTQ）**，因为无法重新训练模型。

#### 主流 PTQ 工具

**NVIDIA TensorRT Model Optimizer (ModelOpt)**：
- 开源库，同时支持剪枝、蒸馏、稀疏性
- 输出兼容所有主流推理引擎（vLLM、SGLang、TensorRT-LLM）

#### 量化决策矩阵

选定精度后，进行训练后量化前需要做两个决策：

```
问题 1：模型的哪些部分需要量化？
问题 2：哪种数值格式能提供合适的动态范围和粒度？
```

#### 模型组件量化敏感度

| 组件 | 敏感度 | 说明 |
|------|--------|------|
| **权重**（特别是线性层）| 🟢 低 | 对量化最不敏感 |
| **激活**（激活函数的中间输出）| 🟡 中 | 对量化敏感度中等 |
| **KV 缓存**（注意力计算的缓存结果）| 🟡 中 | 对量化敏感度中等 |
| **注意力层**（如 softmax）| 🔴 高 | 对量化高度敏感 |

**细化策略**：
- 即使是通常最不敏感的线性层和激活，**输入/输出层**等早期和晚期层也应保留原始精度
- **KV 缓存量化**能增强前缀缓存、解耦等技术的效果（内存中存储更多内容，读取速度更快）
- 但 KV 缓存的量化误差会**逐 token 累积**，因为每个 token 的 KV 缓存会被后续所有 token 使用

> ⚠️ **注意力层量化风险最高**：不仅对动态范围极度敏感，而且每次注意力计算都依赖之前所有注意力计算的结果，误差会在数千 tokens 的序列中快速累积。

#### 温和的量化方案（推荐）

```
使用 FP8 / MXFP8 这类高动态范围格式
谨慎量化部分线性层、激活值
量化常用的 KV 缓存数值
注意力层组件几乎不量化（除最激进的方案外）
```

### 2.3 衡量质量影响

> **生产可用量化的标准**：无感知质量损失。

#### 三种质量检查方法

| 方法 | 说明 | 推荐度 |
|------|------|--------|
| **困惑度（Perplexity）** | 计算量化模型的困惑度分数，与原始模型对比 | ⭐⭐⭐ 最简单 |
| **智能基准** | 运行 MMLU、SWE-bench 等标准基准，对比原始分数 | ⭐⭐⭐⭐ 较全面 |
| **自定义评估** | 在量化模型上运行产品专属评估套件，与原始权重对比 | ⭐⭐⭐⭐⭐ 最全面 |

**困惑度检查原理**：
- 困惑度越高 = 模型对序列越"意外"（这不是一个预测 token 的模型应有的表现）
- 量化后，你希望困惑度增幅极小
- LLM 是非确定性的，分数每次运行都会略有浮动

> 💡 **记住**：量化是一个尺度，不是二元选择。可以通过量化到 FP8 而非 FP4，或者只量化模型更少组件，在降低质量风险的同时获得部分性能提升。

**如果不能承担质量风险**：本文介绍的除量化技术外的所有其他技术（投机解码、缓存、并行、解耦）在质量上都是**无损的**。

## 3. 投机解码（Speculative Decoding）

### 背景

LLM 推理的解码阶段是一个**自回归过程**，token 一次生成一个。解码的瓶颈是**内存带宽**，在低到中等批处理大小下，计算资源会因为等待权重从内存读取而闲置。

**投机解码的核心思想**：利用这些空闲计算资源，尝试在目标模型的单次前向传播中生成多个 tokens。

> ✅ **效果**：如果推理引擎能在每次内存权重往返中生成 2 个、3 个甚至更多 token，每秒生成的 token 数（TPS）会大幅提升。
> 
> ⚠️ **局限性**：投机解码只提升 TPS/ITL，不提升 TTFT。

### 通用机制

```
┌─────────────────────────────────────────────────┐
│              投机解码通用流程                     │
├─────────────────────────────────────────────────┤
│  1. 草稿模型生成 N 个草稿 token                  │
│  2. 目标模型对这些 token 进行验证                 │
│  3. 目标模型接受所有合法草稿 token                │
│  4. 目标模型自行生成 1 个额外 token               │
│  5. 每次前向传播能生成 N+1 个 tokens             │
└─────────────────────────────────────────────────┘
```

> 🎯 **类比**：想象数独游戏——解题很难，但检查答案是否正确非常容易。对目标模型来说，生成 token 就像解数独，验证草稿 token 就像检查已完成的数独。

### 性能影响因素

投机解码的性能提升取决于三个因素：

| 因素 | 说明 | 影响 |
|------|------|------|
| **草稿 token 成本** | 生成一个草稿 token 所需时间 | 成本越低越好 |
| **草稿序列长度** | 每次前向传播生成的草稿 token 数 | 越长越好（但受接受率限制）|
| **token 接受率** | 被目标模型接受的草稿 token 百分比 | 越高越好 |

> 📉 **重要规律**：草稿序列早期的 token 接受率很高，但越往后可靠性越低。一旦单个草稿 token 被判定错误，序列中所有后续 tokens 也会被拒绝。

### 影响接受率的因素

| 因素 | 影响 |
|------|------|
| **温度（Temperature）** | 更高的温度会产生更难预测的 token 分布，降低投机解码效果 |
| **草稿模型匹配度** | 如果草稿模型在数学上比目标模型更熟练于特定主题，接受率会提高 |
| **批处理大小** | 在高批处理大小下，计算资源过于饱和，无法承担验证开销，投机必须动态关闭 |

### 3.1 草稿-目标投机解码（Draft-Target）

**机制**：使用两个模型

```
草稿模型（Draft Model）
  ↓ 生成 N 个草稿 token
目标模型（Target Model）
  ↓ 验证草稿 token，接受合法的，自行生成 1 个
输出：N+1 个 token / 每次前向传播
```

**草稿模型选择原则**：
- ✅ 高 token 接受率
- ✅ 运行资源消耗极小
- ✅ 通常是与目标模型同系列的更小模型（共享分词器和行为）
- 📏 **经验法则**：草稿模型的参数规模应至少比目标模型**小 10 倍**

> 💡 微调或蒸馏可以通过让小模型学习更像目标模型的行为，提升 token 接受率。

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 快速开箱即用，不需要任何训练或微调 | 引入了所有投机解码方法中最大的开销 |
| 草稿模型和目标模型解耦，灵活 | 推理引擎必须在内存中存储草稿模型的权重、激活值和 KV 缓存 |
| | 需要额外的预填充计算周期 |
| | 草稿模型和目标模型必须协调运行，避免资源竞争 |

### 3.2 Medusa

Medusa 是最早替代草稿-目标投机解码的方法之一。

**核心创新**：通过微调目标模型，使其在每次前向传播中生成额外 token，解决了运行草稿模型带来的复杂性和开销。

```
普通 LLM 架构：1 个解码头
Medusa 架构：  1 个原始解码头 + 2~4 个 Medusa 头
```

**工作流程**：
1. Medusa 头在目标模型生成的 token 之上，各自生成一个草稿 token
2. 草稿 token 在下一次前向传播中验证

**现状**：Medusa 在草稿 token 数量和接受率上仍然受限，如今在生产中使用并不广泛。但它启发了 EAGLE 等更流行的技术。

### 3.3 EAGLE

#### 为什么需要 EAGLE？

使用现成预训练模型作为草稿模型的主要问题是：Qwen 0.5B 这类模型被设计为在廉价硬件上运行的优秀独立 LLM，而不是在 B200 上生成投机草稿 token。这些草稿模型运行效率低，接受率相对较低。

#### EAGLE 的解决方案

EAGLE（Extrapolation Algorithm for Greater Language-model Efficiency）提供了一个专用草稿模型：

| 特性 | 说明 |
|------|------|
| **输入** | 以隐藏状态（hidden states）为输入 |
| **输出** | 生成推测 token |
| **草稿 token 数** | 最多 8 个（是 Medusa 的两倍多）|
| **接受率** | 非常高 |
| **参数规模** | 通常小于 10 亿参数，在增加训练数据时扩展性良好 |

> 🔑 **关键创新**：在推理过程中，LLM 会以隐藏状态形式在各层之间累积大量关于预测 token 的上下文信息。传统草稿模型无法访问这些信息，而 EAGLE **以隐藏状态为输入**。

**EAGLE 训练细节**：
- 训练于三组隐藏状态：早期层、中层、后期层各一组
- 通常小于 10 亿参数
- 在增加训练数据时扩展性良好

**工程优势**：
- EAGLE 可以附加到与目标模型相同的模块（PyTorch 类）上
- 每次前向传播都会同时在目标模型和 EAGLE 推测器上运行推理
- 统一流水线解决了草稿-目标解码的多次 CPU 往返编排问题

> 📌 **适用场景**：具备训练 EAGLE head 知识和能力的推理工程师的通用首选推测算法，并得到推理引擎的良好支持。

**权衡**：与其他推测技术一样，为提升延迟采用 EAGLE 需要降低批处理大小，从而降低吞吐量、增加成本。

### 3.4 N-gram 推测与 Lookahead 解码

#### N-gram 推测

**机制**：没有草稿模型，而是使用 N-gram 字典

```
1. 在生成 KV 缓存的同时，推理引擎构建一个 N-gram 字典
2. N-gram 字典将单个起始 token 映射到观察到的 N 个 tokens 序列
3. 解码时，生成的 token 被送入字典，任何可用后缀都会被选为草稿 token
4. 目标模型在下一次前向传播中验证这些草稿 tokens
```

**N-gram 字典构建时机**：在预填充阶段首次构建，包含输入文本中的常见序列。

**优缺点**：

| 优点 | 缺点/局限 |
|------|----------|
| 序列可以远长于 EAGLE（超过 10 个 tokens）| 接受率只在模型输出内容与输入高度相似时才高 |
| 不需要训练草稿模型 | 主要用于代码补全、代码修改等场景 |

**最佳应用场景**：代码补全、代码生成等，语法可预测、输出与输入高度接近。

#### Lookahead 解码

与 N-gram 推测类似，但在推理过程中**生成** N-gram 来填充字典。

| 特性 | N-gram 推测 | Lookahead 解码 |
|------|-------------|----------------|
| 字典来源 | 输入文本 | 推理过程中生成 |
| 通用性 | 低（依赖重复上下文）| 高 |
| 计算开销 | 低 | 高（需要额外计算生成 N-gram）|

**总结**：
- **N-gram 推测**在代码补全等任务中表现出色
- **Lookahead 解码**在计算资源富余的系统中更通用
- 所有投机解码算法都旨在**减少生成完整输出序列所需的总前向传播次数**

---

## 4. 缓存（Caching）

### 背景

在预填充阶段，推理引擎为输入序列构建 **KV 缓存**（每个 token 的键值存储）。然后在解码阶段为每个新生成的 token 更新 KV 缓存。

> ⚠️ **如果没有 KV 缓存**，LLM 推理会慢得无法忍受，因为每个后续 token 都需要重新计算序列中所有先前值。

### KV 缓存复用的

所有推理引擎默认**按请求**使用 KV 缓存。但工程师可以通过**在请求之间复用 KV 缓存**，从 KV 缓存中获得更大价值。

---

### 4.1 前缀缓存与 KV 缓存复用

#### 原理

```
输入序列 1: "Weather in" + "Seattle?"
输入序列 2: "Weather in" + "San Francisco?"

共享前缀: "Weather in"（2 个 tokens）

使用前缀缓存：
  - 第二个请求可以复用第一个请求的 KV 缓存
  - 跳过前 2 个 tokens 的预填充
  - 直接读取已有 KV 缓存 → 提升 TTFT
```

> 💡 **这就是为什么按 token 付费的 API 对"缓存命中"的输入 tokens 收费低于"缓存未命中" tokens**——复用缓存 tokens 几乎不消耗计算和时间。

#### 前缀缓存的最佳应用场景

| 场景 | 说明 |
|------|------|
| **复杂系统提示** | 智能体、面向客户的聊天机器人、RAG 框架、工具调用通常在每次调用时使用冗长复杂的系统提示 |
| **代码补全** | 代码补全、代码生成等编程功能需要传递数千行相同代码作为共享上下文 |
| **文档与检索** | 文档摘要、问答、检索都会在用户提示前添加重复上下文 |
| **多轮对话** | 普通对话会在对话模板中重复所有历史消息，每轮对话都能从前缀缓存中节省更多时间 |

#### 前缀缓存的局限性

前缀从序列开头一直工作到**第一个不重复 token**。

```
示例：
序列 1: [A][B][C][D]
序列 2: [A][B][X][Y]

共享前缀: [A][B]（2 个 tokens）
→ 缓存只能复用 [A][B]，[X] 虽然和 [C] 不同，但缓存到此结束
```

> ⚠️ **重要**：因为 LLM 是自回归的，每个 token 都会影响后续所有 tokens。因此单个新异 token 会改变模型内部对序列其余部分的表示。

**优化技巧**：要利用前缀缓存，请确保**新异 tokens 尽可能放在上下文末尾**。

```
❌ 不好的提示方式：
  "User: 今天天气怎么样？\nAssistant:" + [历史对话]

✅ 好的提示方式：
  [历史对话] + "User: 今天天气怎么样？\nAssistant:"
```

#### 非前缀 KV 缓存复用（前沿研究）

围绕其他类型 KV 缓存复用的研究正在积极进行：

| 工具 | 说明 |
|------|------|
| **CacheBlend** | 支持非前缀序列的 KV 缓存复用 |
| **LMCache** | 支持非前缀序列的 KV 缓存复用 |

这些方法需要校正位置嵌入，并选择性地重新计算 KV 条目以保持输出质量。

---

### 4.2 KV 缓存存储位置

KV 缓存非常宝贵，但占用大量内存，而 GPU 的 VRAM 有限。

#### 内存分配配置示例（TensorRT-LLM）

```python
# 在 TensorRT-LLM 中配置 KV 缓存内存分配
kv_cache_config = KVCacheConfig(
    free_gpu_memory_fraction=0.8  # 将剩余 VRAM 的 80% 分配给 KV 缓存
)
```

**示例计算**：
- B200 GPU：180GB VRAM
- 模型权重和缓冲区：100GB
- 剩余 VRAM：80GB
- KV 缓存分配：80GB × 80% = **64GB**

> ⚠️ **一旦分配空间用尽**（而且很快会用尽），你就必须开始删除已保存的 KV 缓存，增加未来请求缓存未命中的概率。

#### KV 缓存的四级存储

为了获得更多 KV 缓存空间，可以从 VRAM 卸载到其他就近存储：

| 级别 | 内存类型 | 近似速度 | 近似容量 | 适用场景 |
|------|---------|---------|---------|---------|
| **G1** | 设备内存（GPU VRAM）| 每秒 TB 级 | 几十到几百 GB | 热缓存 |
| **G2** | 主机内存（CPU RAM）| 每秒几十到几百 GB | 几百 GB 到 TB 级 | 温缓存 |
| **G3** | 本地 SSD | 5~10 GB/s | TB 级 | 冷缓存 |
| **G4** | 网络 SSD | 每秒 GB 级 | 几十 TB | 全局缓存 |

> 💡 **GB200 等型号**配备 CPU 和高速互联，非常适合 KV 缓存卸载。

#### NVIDIA Dynamo 的 KVBM

NVIDIA Dynamo 通过 **KVBM（KV 块管理器）** 支持 KV 缓存卸载：

```python
# KVBM 提供 API，用于在不同内存级别之间移动 KV 缓存块
kvbm = KVBlockManager(
    levels=["gpu", "cpu", "ssd"],  # 定义存储层级
    policy="lru"                   # 替换策略：最近最少使用
)
```

**最佳实践**：将最常使用的块放在高带宽内存（G1）中，不常用的块可以放到慢速存储（G2/G3/G4），直到需要时再读取。

---

### 4.3 缓存感知路由

#### 问题

在生产环境中，推理服务会有多个副本，传入流量会在副本间分配。通常，流量路由基于每个副本的繁忙程度。

#### 缓存感知路由的原理

```
传统路由：基于每个副本的繁忙程度均匀分配请求
缓存感知路由：基于 KV 缓存分配流量
```

**效果**：
- 进行长时间对话聊天或多次询问代码库问题的用户，应尽可能将请求路由到同一个副本
- 获得缓存命中 → 请求更快、成本更低

#### 全局 KV 缓存

另一种选择是使用 **G4 级别的网络存储**构建跨副本的全局 KV 缓存：

```
优点：
  - 所有副本最终都能访问任何预计算序列
  - 在节点循环或自动扩缩容下线时不会丢失缓存序列

仍然重要的路由：
  - 拥有 G1 级别热缓存的副本会比从 G4 读取的副本更快处理请求
```

---

### 4.4 长上下文处理

#### 什么是"长上下文"？

"长上下文"这个说法在定义上有点同义反复：**当序列生成的 KV 缓存大到在推理中引发问题时，它就成为"长上下文"**。

根据模型、硬件、引擎和流量的不同，这些问题在超过 **32K、64K 或 128K tokens** 等常见阈值后开始出现。

#### 长上下文带来的挑战

考虑到 KV 缓存，注意力方程随序列长度**线性增长**。在长序列下：

```
注意力成为 VRAM 的主要消耗者
而 VRAM 正是解码的瓶颈
```

#### 解决方案

**模型级解决方案**（逐个模型基础）：
- 滑动窗口注意力
- 压缩注意力
- 稀疏注意力

**通用优化方法**：

| 技术 | 说明 |
|------|------|
| **FlashAttention** | 一系列优化的注意力内核，以更少的内存读写计算注意力 |
| **PagedAttention** | 一种内存管理技术，将 KV 缓存存储在固定大小的页面中，减少碎片和重复 |
| **分块预填充** | 将长输入序列切分成块的策略，可以在资源允许时与解码同时运行，避免长序列压垮推理引擎 |

> 🚨 **如果经过这些优化后，仍然需要超过单块 GPU 所能提供的 VRAM 来存储 KV 缓存**：你需要跨多 GPU 并行推理（参见下一节"模型并行"）。

---

## 5. 模型并行（Model Parallelism）

### 背景

目前市面上所有前沿 LLM 都大到无法在单块 GPU 上运行批处理推理。

> 📊 **简单估算**：在 FP8 精度下，加载 10 亿（1B）参数模型权重大约需要 **1GB VRAM**。

**实际案例**：
- DeepSeek-V3.1：6710 亿（671B）参数
- 仅模型权重：FP8 下约 671GB
- 单块 B200 GPU：240GB VRAM → **直接 OOM（Out Of Memory）**

> 💡 **即使勉强把模型权重塞进 VRAM 也是不够的**。在 4 块 B200 GPU（960GB VRAM）上，理论上可以加载 DeepSeek 权重。但没有剩余空间留给 KV 缓存（通常在加载权重后占用 80% 以上的剩余 VRAM）。

**结论**：需要整整 8 块 B200 的一个节点来为 DeepSeek 这种规模的模型提供真实生产流量。

```
估算所需最小 GPU 数量：
  (模型参数量 × 精度字节数 + 预期 KV 缓存大小)
  ÷ 单 GPU VRAM
  = 向上取整到下一个可用实例大小
```

---

### 并行推理的限制

所有这些都要求推理能高效地从单卡扩展到多卡。并行推理扩展的限制是 **GPU 间通信开销**。

#### GPU 互联带宽对比

| 互联方式 | 带宽 | 适用场景 |
|---------|------|---------|
| **VRAM** | 每秒 TB 级 | 单 GPU 内部 |
| **NVLink / NVSwitch** | 每秒几百 GB ~ TB 级 | 节点内 GPU 间 |
| **InfiniBand** | 每秒几十 ~ 几百 GB | 节点间 |

> ⚠️ NVLink 和 InfiniBand 提供高带宽，但它们的速度**远低于 VRAM**。由于解码受内存带宽限制，多 GPU 推理必须精心设计，避免 GPU 间通信成为瓶颈。这一研究领域称为**拓扑感知并行**。

---

### 三种主要模型并行形式

| 方法 | 全称 | 机制 | 缺点 |
|------|------|------|------|
| **PP** | Pipeline Parallelism（流水线并行）| 将模型层切分到不同 GPU | 不推荐用于推理：延迟和利用率差，分步流水线 |
| **TP** | Tensor Parallelism（张量并行）| 将每层内的张量切分到不同 GPU | 需要 GPU 间同步，不适合多节点 |
| **EP** | Expert Parallelism（专家并行）| 将 MoE 模型的整个专家分片到不同 GPU | 需要 GPU 间路由访问多个专家，更适合吞吐量 |

> 📌 **推荐**：张量并行通常最适合单节点内低延迟模型推理，专家并行提升 MoE LLM 的吞吐量。流水线并行仅用于多节点推理。

---

### 5.1 低延迟张量并行（TP）

#### 原理

张量并行将模型的**每一层**拆分（与流水线并行保留完整层不同），并将层片段分布到分配的 GPU 上。

```
张量并行工作流程：

GPU 0: 存储层 1 的部分权重 + 层 2 的部分权重...
GPU 1: 存储层 1 的部分权重 + 层 2 的部分权重...
...
GPU N: 存储层 1 的部分权重 + 层 2 的部分权重...

每层的结果需要以归约方式合并为单个输出
→ 然后才能计算下一层
```

#### 通信开销

每层的结果需要以归约（reduce）方式合并为单个输出，然后才能计算下一层。

- ✅ **节点内**：具备高带宽 NVLink 和 NVSwitch 的节点中，这种通信开销被最小化
- ❌ **多节点**：InfiniBand 速度远低于 NVLink，通信开销成为瓶颈

#### 适用场景

| 场景 | 适用性 |
|------|--------|
| 稠密模型（如 Llama 405B）| ✅ 适合 |
| MoE 模型 | ✅ 适合 |
| 单节点内 | ✅ 推荐 |
| 多节点 | ❌ 不推荐（通信开销过大）|

> 💡 **增加张量并行度能提升单用户 TPS**（假设模型足够大、序列足够长，通信开销不会超过更快的前向传播）。

---

### 5.2 高吞吐量专家并行（EP）

#### 原理

专家并行将专家**干净地**分到不同 GPU。

```
示例：128 专家的模型，以 EP8 跨 8 块 GPU 运行
  GPU 0: 托管专家 0~15（16 个专家）
  GPU 1: 托管专家 16~31（16 个专家）
  ...
  GPU 7: 托管专家 112~127（16 个专家）
```

#### 优缺点

| 优点 | 缺点/局限 |
|------|----------|
| 提升系统总吞吐量 | 需要 GPU 间路由访问多个专家 |
| 推理更具可扩展性、成本更低 | 更适合吞吐量优化，而非延迟优化 |
| 单个专家独立处理 token，每个 token 耗时相同 | |
| 系统整体能处理更多并发 token | |

#### 通信开销对比

| 并行方式 | 通信内容 | 通信开销 |
|---------|---------|---------|
| **张量并行（TP）** | 每层结果需要收集归约 | 高 |
| **专家并行（EP）** | token 在专家间传递 | 低（专家路由器很小，复制到每个 GPU）|

> 💡 **由于通信开销更低，EP 能很好地扩展到多节点部署和互联带宽有限的系统。**

#### TP + EP 混合使用

许多部署混合使用 TP + EP，同时获得两种优势：

```
混合部署示例：
  - 对注意力层使用 TP（张量并行）
  - 对稀疏 MoE 层使用 EP（专家并行）
```

---

### 5.3 多节点推理

#### 何时需要多节点推理？

- 服务超大规模模型（如 DeepSeek-V3.1 671B）
- 高精度（如 FP16 而非 FP8）
- 支持数百万 token 输入序列
- 希望尽可能快地运行推理

#### 多节点推理的挑战

| 挑战 | 说明 |
|------|------|
| **基础设施** | 如何可靠地申请两个或更多互联 GPU 节点，并跨云厂商构建抽象？ |
| **并行策略** | 如何在速度远低于 NVLink 的 InfiniBand 上有效通信？ |

#### 多节点并行策略推荐

| 模型类型 | 推荐策略 | 说明 |
|---------|---------|------|
| **稠密模型** | 节点内 TP + 节点间 PP | 如 TP8PP2（8 卡张量并行 + 2 节点流水线并行）|
| **MoE 模型** | EP（专家并行）| 通信开销比张量并行低，更适合多节点 |
| **延迟优先** | TP8PP2 | 更低的单用户延迟 |
| **吞吐量优先** | EP16 | 更高的系统总吞吐量 |

> ⚠️ **除非你的模型和 KV 缓存大到必须多节点推理，否则这可能不是额外硬件的最佳使用方式**。你通常**最好将额外节点用于副本水平扩展，或解耦服务**。

## 6. 解耦（Disaggregation）

### 核心思想

解耦结合了推理工程中的三个重要思想：

```
1. 预填充是决定 TTFT 的计算密集型过程
   解码是决定 TPS 的内存密集型过程
   → 两个阶段对资源的需求不同！

2. 专业化能提升从内核选择到推理引擎参数调优的所有环节性能

3. 如果能避免低带宽互联带来的瓶颈，你可以在多 GPU 甚至多节点上
   有效扩展模型服务
```

> 💡 **问题**：当预填充和解码在高流量下运行在同一节点时，它们相互干扰的概率更高。理想情况下，预填充使用更多计算资源，解码使用更多内存，两者可以高效共存。但是，在更大批处理和更多计算密集型优化下，预填充和解码会开始竞争资源。

---

### 6.1 解耦工作原理

#### 什么是解耦？

解耦（或称解耦服务）是将预填充和解码分离到不同 GPU / 节点上的独立引擎的思想。

#### 解耦推理流程

```
┌─────────────────────────────────────────────────────────────┐
│                    解耦推理三步流程                           │
├─────────────────────────────────────────────────────────────┤
│  步骤 1：预填充引擎接收输入序列，                             │
│          生成 KV 缓存并计算第一个 token                        │
│                        ↓                                     │
│  步骤 2：预填充引擎通过硬件互联                               │
│          将 KV 缓存发送到解码引擎                             │
│                        ↓                                     │
│  步骤 3：解码引擎计算所有后续 tokens                           │
└─────────────────────────────────────────────────────────────┘
```

#### 条件解耦（推荐）

在条件解耦中，请求首先发送到解码引擎，它会检查输入序列是否已缓存或足够短，可以本地处理：

```
if 输入序列已缓存 OR 序列长度 < 阈值:
    解码引擎本地处理预填充（跳过解耦）
else:
    解码引擎将请求转移到预填充引擎进行解耦服务
```

> ✅ **条件解耦更适合真实世界流量**，避免对短序列或缓存命中请求引入不必要的解耦开销。

#### 解耦的优势

将预填充和解码分离后，你可以**分别优化每个引擎**：

| 引擎类型 | 资源需求 | 推荐配置 |
|---------|---------|---------|
| **预填充引擎** | 计算密集型 | 需要更低的 TP（张量并行度）|
| **解码引擎** | 内存密集型 | 需要更高的 TP 或更大的 KV 缓存空间 |

---

### 6.2 何时使用解耦

解耦非常强大，但需要多 GPU 和额外工程工作。

#### 使用解耦的三个条件

| 条件 | 说明 | 阈值参考 |
|------|------|---------|
| **条件 1** | 服务大流量 | 每天 1 亿～10 亿 tokens 起 |
| **条件 2** | 服务大型模型 | 至少 100B 参数 |
| **条件 3** | 流量是预填充密集型 | 输入序列长 |

#### 决策树

```
是否满足 条件 1 或 条件 2？
  ├─ 否 → 不要在额外硬件上浪费钱，性能提升极小
  └─ 是 ↓
是否满足 条件 3？
  ├─ 否 → 最好用额外 GPU 水平扩展副本
  │         （解码引擎对短序列或前缀缓存命中更高效）
  └─ 是 → ✅ 适合使用解耦
```

#### 解耦的绝佳用例

> 📝 **教科书式工作负载**：在代码编辑器中服务前沿 LLM，许多开发者同时传入大量多样的代码块作为上下文。
> 
> **特征匹配**：大量 tokens、以预填充为主、万亿参数 LLM。

---

### 6.3 基于 NVIDIA Dynamo 的动态解耦

[NVIDIA Dynamo](https://github.com/ai-dynamo) 为解耦提供生产就绪支持，具备处理异构真实流量的灵活性。

#### Dynamo 提供的关键功能

| 功能 | 说明 |
|------|------|
| **预填充队列** | 在所有预填充引擎饱和时保存请求 |
| **条件解耦支持** | 基于前缀缓存后 ISL 阈值和预填充队列大小路由预填充 |
| **高效 KV 传输** | 基于 NIXL 的高效 KV 从预填充传输到解码引擎；当引擎有不同 TP 配置时，使用内核转置 KV 块布局 |

#### 动态解耦

这些功能共同实现**动态解耦**：

```
预填充和解码引擎数量可在运行时配置
并随传入流量变化动态调整
```

#### xPyD 表示法

解耦不需要预填充与解码引擎 1:1 比例。表示法：`xPyD`

```
示例：5P3D
  = 5 个预填充引擎 + 3 个解码引擎
  = 协同服务单个模型部署
```

#### 潜在瓶颈与解决方案

| 瓶颈 | 原因 | 解决方案 |
|------|------|----------|
| **预填充队列过长** | 预填充引擎饱和 | 动态调整 xPyD，为预填充分配更多资源；设置合理的队列大小阈值 |
| **解码引擎 KV 缓存不足** | 高负载下 KV 缓存空间不够 | 通过量化和 KV 缓存卸载增加 KV 缓存可用空间 |

## 总结

本文介绍了大模型推理加速的五类关键技术：

```
┌─────────────────────────────────────────────────────────────┐
│                  推理加速技术总结                              │
├──────────┬──────────────────────────────────────────────────┤
│ 量化      │ 降低数值精度，提升延迟和吞吐量，可能损失质量          │
│ 投机解码  │ 利用空闲计算生成多个 token，提升 TPS                │
│ 缓存      │ 复用 KV 缓存，降低 TTFT 和 TPS                    │
│ 模型并行  │ 跨多 GPU 分布模型，支持大模型推理                    │
│ 解耦      │ 分离预填充和解码，分别优化                          │
└──────────┴──────────────────────────────────────────────────┘
```

### 技术组合建议

| 场景 | 推荐技术组合 |
|------|-------------|
| **中小模型 + 低流量** | 量化 + 前缀缓存 |
| **大模型 + 中等流量** | 量化 + 前缀缓存 + 投机解码（EAGLE）|
| **大模型 + 高流量 + 长上下文** | 量化 + 前缀缓存 + 投机解码 + 模型并行（TP/EP）|
| **超大模型 + 超高流量 + 长上下文** | 全部技术 + 解耦（动态 xPyD）|
